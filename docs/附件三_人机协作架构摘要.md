# 附件三：人机协作架构摘要

> 来源：AgentNexus_概要设计说明书_v2.0_附件三_人机协作架构.docx

## 1. 协作理念与设计原则

- 将「对话式交互」升级为「协同式作业」：人类决策意图实时同步给 AI；AI 执行成果即时反馈并更新到项目知识。
- 分工边界：人类负责顶层规划、方向决策、关键节点确认（Human-in-the-loop）；Agent 负责复杂执行、过程沉淀、低风险任务自主推进。

## 2. 人类意图的实时同步

- **问题**：主文档中人类更新意图需进后台编辑 ANCHOR.md，为静态、手动流程。
- **意图触发指令**：在聊天频道引入轻量指令，无需离开对话即可更新 AI 认知基准。
  - `/anchor 更新目标`：立即更新 ANCHOR.md，通知频道内所有在线 Bot 刷新上下文。
  - `/decide [结论内容]`：立即追加到 DECISIONS.md。
  - `/redirect @bot [新指令]`：打断当前 Bot 正在执行的任务，注入新指令。
  - `/pause`：暂停当前频道所有 Bot 待执行任务队列。
  - `/resume`：恢复暂停的任务队列。
  - `/status`：Bot 输出当前任务进度、已完成步骤、下一步计划。
- **知识确认机制**：Bot 自动向 DECISIONS.md 追加决策时，应向频道发送确认请求（👍 确认 | ✏️ 修改 | 🗑️ 删除）；48 小时内无响应可默认保留但标记「待确认」。

## 3. 任务生命周期与 Human-in-the-loop

- **Task 概念**：引入 Task 作为核心抽象，含 task_id、title、channel_id、owner_bot、status（pending/running/awaiting_human/completed/failed/cancelled）、risk_level、steps、created_by、时间戳等。
- **两种模式**：主动确认模式（任务完成一阶段后 Bot 主动停下汇报并请求确认）；被动拦截模式（检测到高风险操作时审批或拒绝）。完全自主模式：低风险步骤自动执行并推进。

## 4. 自主协作执行机制

- **Critique Loop**：配置执行者 Agent 与评审者 Agent，执行完一步后评审产出物是否符合 ANCHOR、是否存在错误、是否满足用户意图；SUCCESS 则自动触发下一步，FAIL 则回传 Executor 修正，超过 max_critique_retries 则上报人类。
- **自主错误恢复**：执行失败时先将错误回传 Executor 自我修正；连续失败超过 max_error_retries 则任务设为 awaiting_human，在频道发送错误报告。
- **Task DAG**：任务步骤为有向无环图，每节点有独立风险等级；低风险可完全自主，中风险需 Critique 评审，高风险需人类确认后继续。

## 5. 风险等级定义

- 低风险（low）：可逆、影响局限，完全自主执行。
- 中风险（medium）：影响正式文件但可回滚，执行 + Critique Loop。
- 高风险（high）：不可逆或影响外部系统，执行后强制等待人类确认。

## 6. 前端 UX 补充

- 任务进度面板、确认卡片、知识确认气泡、意图指令快捷栏、错误报告卡片、自主执行进度条。

## 7. 当前实现对齐（2026-03）

- 当前已上线的人机协作能力，核心是 `@channel bot`、澄清卡片、建议接手与任务链路追踪。
- 摘要中提到的 `/anchor`、`/decide`、`/pause`、Task DAG 可视化、完整 Critique Loop 目前仍属规划能力。
- 对外说明时应避免将规划能力描述为“默认已上线”。
