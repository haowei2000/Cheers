# AgentNexus 系统管理说明书

> 面向**系统管理员**：如何创建工作空间与项目、如何添加/移除成员、如何把 OpenClaw 接入为 Bot 等。需要会使用命令行与接口（API）。

---

## 管理员入口地址（当前 M1）

当前版本**没有单独的管理员后台页面**，管理员与普通用户使用同一前端；管理操作通过**后端 API** 完成。

| 入口 | 地址 |
|------|------|
| **聊天前端**（与用户相同） | Docker：`http://localhost:80`；本地开发：`http://localhost:5173` |
| **API 文档（Swagger）** | **`http://localhost:8000/docs`**（创建频道、添加成员、查看列表等均在此调用或参考） |
| **健康检查** | `http://localhost:8000/health` |
| **OpenClaw 发现与注册指南（机器可读）** | `http://localhost:8000/api/public/agentnexus-discovery` |

若后端端口或域名不同，请将上述 `localhost:8000` 替换为实际后端地址。

### 管理界面中的「表格表单」模式

当前管理操作（创建项目、创建 Bot、添加/移除成员、**待审核 Bot 申请** 等）均采用**同一套交互方式**：在「管理」弹窗中，**左列是字段说明/问题，右列是对应的输入框或操作按钮**，用户填写后点击提交即可完成。后续新增的系统设置（如更多工作空间配置、频道配置等）也会沿用这种「表格表单」方式，便于统一体验与扩展。

## 一、概念说明

| 概念 | 说明 |
|------|------|
| **工作空间（Workspace）** | 顶层容器，相当于“组织”。创建项目前必须先有一个工作空间。 |
| **项目（频道 / Channel）** | 协作单元，成员在项目里聊天、@ Bot、发文件。 |
| **成员** | 分为 **user**（用户）和 **bot**（Bot）；均需被“加入”到具体项目后，才能在该项目中生效。 |
| **Bot 账号** | 每个 OpenClaw（或其它 AI 服务）在系统里对应一个 Bot 账号；注册 Bot 后再把它加入项目，用户才能 @ 它。 |

## 二、如何创建工作空间与项目

### 2.1 创建工作空间（只需做一次）

**【推荐】前端表格表单**：在「管理」→「创建工作空间」中填写「空间名称」后点击「创建」即可。

**或**在数据库中插入一条工作空间记录。

**Docker 部署：**

```bash
docker compose exec backend sqlite3 /app/data/main.db "INSERT INTO workspaces (workspace_id, name, created_at) VALUES ('ws-default-001', '默认空间', datetime('now'));"
```

**本地部署：**

```bash
cd backend
sqlite3 ../data/main.db "INSERT INTO workspaces (workspace_id, name, created_at) VALUES ('ws-default-001', '默认空间', datetime('now'));"
```

记下 **`workspace_id`**（如 `ws-default-001`），创建项目时要用。

### 2.2 创建项目（频道）

在已有工作空间下创建项目，调用 API（基础地址默认 `http://localhost:8000`，下同）：

```bash
curl -X POST http://localhost:8000/api/channels \
  -H "Content-Type: application/json" \
  -d '{"workspace_id":"ws-default-001","name":"我的第一个项目","type":"public","purpose":"日常协作"}'
```

将 **`ws-default-001`** 换成你的工作空间 ID，**name** 可改为任意项目名称。  
返回的 JSON 中的 **`channel_id`** 即为项目 ID，后续加人、加 Bot 时需用到。

## 三、如何管理项目成员（加人 / 加 Bot / 移除）

### 3.1 查看项目成员

```bash
curl "http://localhost:8000/api/channels/项目ID/members"
```

将 **项目ID** 替换为实际的 `channel_id`。

### 3.2 添加成员（用户或 Bot）

**方式一：聊天内邀请 Bot（Slack 风格，推荐）**

- 用户在频道内输入 **@** 时，会显示全部 Bot 列表（已在频道的排前，未加入的灰色排后）。
- 若选择未加入的 Bot，系统提示「是否邀请加入？」；用户确认后，Bot 自动加入该频道，输入框插入 @Bot名，不打断提问流程。
- 无需去「管理」填 ID，适合快速拉 Bot 进群。详见《普通用户使用说明》§3.2.1。

**方式二：管理端添加**

```bash
curl -X POST "http://localhost:8000/api/channels/项目ID/members" \
  -H "Content-Type: application/json" \
  -d '{"member_id":"用户或Bot的ID","member_type":"user"}'
```

- 添加**用户**：`member_type` 填 **`user`**，`member_id` 填该用户的 **user_id**（需先在 users 表存在）。
- 添加**Bot**：`member_type` 填 **`bot`**，`member_id` 填该 Bot 的 **bot_id**（需先在 bot_accounts 表存在，见下文「四、OpenClaw 接入」）。

### 3.3 移除成员

```bash
curl -X DELETE "http://localhost:8000/api/channels/项目ID/members/成员ID"
```

**成员ID** 即该用户或 Bot 的 user_id / bot_id。

## 四、如何让 OpenClaw 接入（注册 Bot 并加入项目）

“接入”指：把一台 OpenClaw 服务注册为 AgentNexus 里的一个 Bot，用户在某项目里 @ 该 Bot 时，由该 OpenClaw 回复。

### 4.1 前提与实现说明

- 当 **openclaw_endpoint** 填写的为 **http://** 或 **https://** 地址时，用户 @ 该 Bot 时，AgentNexus 会向该地址 **POST /execute** 发送请求，并解析返回的 JSON 作为 Bot 回复（请求/响应约定见下文 §4.4）。
- 若 openclaw_endpoint 填的是 `guide://` 或非 http 地址，则不会发起真实 HTTP 调用（引导 Bot 或占位回复）。
- 你的服务需已部署并可访问，并实现 §4.4 的接口约定。

### 4.2 第 1 步：注册 Bot 账号（写入数据库）

当前无 Web 向导，需在 **bot_accounts** 表中插入记录。必填字段示例：

- **bot_id**：唯一 ID，建议 UUID。
- **username**：用户 @ 时使用的名字（如 `mybot`），与显示名可不同。
- **openclaw_endpoint**：OpenClaw 服务地址（必填）。
- **display_name**：显示名称（选填）。
- **status**：如 `online`。

**生成 bot_id（UUID）：**

```bash
python3 -c "import uuid; print(uuid.uuid4())"
```

**Docker 部署时插入 Bot：**

```bash
docker compose exec backend sqlite3 /app/data/main.db "INSERT INTO bot_accounts (bot_id, username, display_name, openclaw_endpoint, status) VALUES ('你的BOT_ID', 'mybot', 'OpenClaw 助手', 'http://你的OpenClaw地址:端口', 'online');"
```

**本地部署时插入 Bot：**

```bash
cd backend
sqlite3 ../data/main.db "INSERT INTO bot_accounts (bot_id, username, display_name, openclaw_endpoint, status) VALUES ('你的BOT_ID', 'mybot', 'OpenClaw 助手', 'http://你的OpenClaw地址:端口', 'online');"
```

将 **你的BOT_ID**、**mybot**、**OpenClaw 助手**、**http://你的OpenClaw地址:端口** 替换为实际值。

### 4.3 第 2 步：将 Bot 加入项目

使用「三、3.2」的添加成员接口，**member_type** 填 **bot**，**member_id** 填上一步的 **bot_id**：

```bash
curl -X POST "http://localhost:8000/api/channels/项目ID/members" \
  -H "Content-Type: application/json" \
  -d '{"member_id":"你的BOT_ID","member_type":"bot"}'
```

完成后，用户在该项目内 @ 该 Bot 的 **username**（如 `@mybot`）即可触发。当 openclaw_endpoint 为 http(s) 时，会真实请求该地址。

### 4.4 真实调用的请求/响应约定（openclaw_endpoint 为 http(s) 时）

AgentNexus 会向 **`{openclaw_endpoint}/execute`** 发起 **POST**，请求体为 JSON：

| 字段 | 说明 |
|------|------|
| task_id | 任务 ID |
| channel_id | 频道 ID |
| trigger_message | 触发消息，含 user、text、timestamp |
| memory_context | 四层记忆键值 |
| attachments | 附件列表 |
| process_config | 如 mode、timeout_seconds |

**响应**需为 JSON，且包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| content | string | Bot 回复正文 |
| success | bool | 是否成功 |
| error_message | string（可选） | 失败时的错误信息 |

若你的 OpenClaw 或自建服务实现上述接口，即可被 AgentNexus 真实调用；否则需做一层桥接。  
**说明**：Bot 的 openclaw_endpoint 填 **http(s)://** 时才会真实调用；填 guide:// 为引导 Bot，填 mock:// 或其它仅为占位回复（演示用），生产环境应使用 http(s) 地址。

## 五、如何让外部 OpenClaw 发现并自动注册（需管理员审核）

本节说明：**外部 OpenClaw 如何发现本系统、获取机器可读的注册指南，并自动提交注册申请**；**管理员在管理界面上审核确认后**，该 Bot 才能真正被加入项目并被 @。

> **快速接入**：详见 [OpenClaw接入指南](OpenClaw接入指南.md)，含完整 curl 示例与常见错误说明。

### 5.1 本系统对外提供的 API / URL

| 用途 | 方法 | URL（基础地址默认 `http://localhost:8000`） |
|------|------|---------------------------------------------|
| **发现与注册指南（OpenClaw 可读）** | GET | `/api/public/agentnexus-discovery` |
| **提交注册申请（OpenClaw 调用）** | POST | `/api/bots/register-request` |

- **发现接口**：`GET {BASE}/api/public/agentnexus-discovery` 返回 JSON，包含本系统名称、注册入口 URL、请求体格式说明等，OpenClaw 可根据此 JSON 自动构造并发送注册请求。**注意：路径必须包含 `/api`，`/public/agentnexus-discovery` 会返回 404。**
- **注册申请接口**：`POST {BASE}/api/bots/register-request` 无需鉴权，请求体为 JSON（见下文），提交后会在系统内生成一条「待审核」申请；**只有管理员在「管理」界面中点击「通过」后，才会真正创建 Bot 账号**，之后管理员可将该 Bot 加入项目，用户才能 @ 使用。

### 5.2 OpenClaw 可读的发现与注册指南（JSON 结构）

OpenClaw 请求 `GET /api/public/agentnexus-discovery` 可获得如下结构的 JSON（示例，字段以实际返回为准）：

| 字段 | 说明 |
|------|------|
| `name` | 系统名称，如 AgentNexus |
| `base_url` | 本系统 API 根地址 |
| `register_request.url` | 提交注册申请的完整 URL |
| `register_request.method` | 固定为 POST |
| `register_request.content_type` | 固定为 application/json |
| `register_request.body_schema` | 请求体字段说明（username、display_name、openclaw_endpoint） |
| `execute_contract` | 审核通过后，系统如何调用 OpenClaw（POST /execute 等）的简要说明 |

**提交注册申请时的请求体示例：**

```json
{
  "username": "mybot",
  "display_name": "OpenClaw 助手",
  "openclaw_endpoint": "https://我的OpenClaw地址:端口"
}
```

- `username`：必填，用户 @ 时使用的名字，系统内唯一。
- `display_name`：选填，显示名称。
- `openclaw_endpoint`：必填，该 OpenClaw 服务的 http(s) 根地址；审核通过后，AgentNexus 会向 `{openclaw_endpoint}/execute` 发送 POST 请求（格式见 §4.4）。

**响应示例（申请已提交、待审核）：**

```json
{
  "status": "success",
  "data": {
    "request_id": "uuid",
    "message": "注册申请已提交，等待管理员在「管理」界面审核通过后可被加入项目并 @。"
  }
}
```

OpenClaw 可根据上述指南实现：发现 AgentNexus → 读取 discovery JSON → 自动 POST 到 `register-request` → 等待管理员审核；审核通过后，管理员在「管理」中将该 Bot 加入具体项目，即可被 @。

### 5.3 注册流程概览

1. **OpenClaw 发现本系统**：请求 `GET /api/public/agentnexus-discovery`，获取 `register_request.url` 与 `body_schema`。
2. **OpenClaw 提交申请**：向 `POST /api/bots/register-request` 发送 JSON（username、openclaw_endpoint 等），系统创建一条状态为「待审核」的申请。
3. **管理员审核**：在聊天前端的「管理」→「待审核 Bot 申请」中看到待审核列表，可对每条申请点击「通过」或「拒绝」。
4. **通过后**：系统自动创建 Bot 账号；管理员再在「添加成员」中将该 Bot 加入目标项目，用户即可在项目内 @ 该 Bot 使用。

**只有经过管理员在管理界面上的审核确认，该 Bot 才会被创建并可被 @；未审核或已拒绝的申请不会产生可用 Bot。**

### 5.4 管理员审核操作

- **入口**：聊天前端左侧「管理」→ 区块「待审核 Bot 申请」。
- **列表**：展示所有 status 为「待审核」的申请（username、openclaw_endpoint、申请时间等）。
- **操作**：每条申请提供「通过」「拒绝」按钮；点击「通过」即创建 Bot 并标记申请为已通过，点击「拒绝」即标记为已拒绝，不再创建 Bot。

## 六、管理操作速查

| 操作 | 方法 | 路径 / 请求体 |
|------|------|----------------|
| 获取所有项目列表 | GET | `/api/channels` |
| 创建项目 | POST | `/api/channels`，body：`workspace_id`, `name`, `type`（如 public）, `purpose`（可选） |
| 获取某项目成员 | GET | `/api/channels/{项目ID}/members` |
| 添加成员（用户或 Bot） | POST | `/api/channels/{项目ID}/members`，body：`member_id`, `member_type`（user / bot） |
| 移除成员 | DELETE | `/api/channels/{项目ID}/members/{成员ID}` |
| 获取频道上下文 | GET | `/api/channels/{项目ID}/context` |
| 更新频道上下文 | PUT | `/api/channels/{项目ID}/context` |
| 创建工作空间 | POST | `/api/workspaces`，body：`name` |
| 获取 Bot 注册申请列表 | GET | `/api/bots/registration-requests`，可选 `?status=pending` |
| 通过 Bot 注册申请 | POST | `/api/bots/registration-requests/{request_id}/approve` |
| 拒绝 Bot 注册申请 | POST | `/api/bots/registration-requests/{request_id}/reject` |
| OpenClaw 发现与注册指南（机器可读） | GET | `/api/public/agentnexus-discovery` |
| 外部提交 Bot 注册申请 | POST | `/api/bots/register-request`，body：`username`, `openclaw_endpoint`, `display_name`（选填） |
| 获取 Orchestrator 配置 | GET | `/api/admin/settings/orchestrator` |
| 更新 Orchestrator 配置 | PUT | `/api/admin/settings/orchestrator`，body：`orchestrator_direct_answer`, `orchestrator_auto_takeover` |
| 资源监控（任务统计） | GET | `/api/tasks/stats?limit_days=7` |

所有接口基础路径：`http://localhost:8000`（或实际后端地址）。更多参数与响应见 **http://localhost:8000/docs**。

### Orchestrator 配置（门户阶段一）

Orchestrator 是系统内置的业务问答 Bot（username=coordinator，bot_id=bot-coordinator-001）。需先将其加入频道后，用户才能 @coordinator 或享受「直接回答」功能。

**【前端入口】** 左侧「管理」→「LLM 设置」→「Orchestrator 配置」：

| 配置项 | 说明 |
|--------|------|
| **直接回答未 @ 的问题** | 开启后，用户发消息未 @ 任何人时，由 Orchestrator 回答业务问题；关闭则必须 @coordinator 才回复 |
| **自动接手** | 开启后，Orchestrator 回复中含「建议 @部门bot 回答 XXX」时，被建议的 Bot 自动接手回答；关闭则需用户手动 @ |

**Orchestrator 使用的 LLM**：在「功能绑定」中为「Orchestrator」选择 LLM；未选时使用「系统 LLM」。

**将 Orchestrator 加入频道**：在「添加成员」中，member_id 填 `bot-coordinator-001`，member_type 选 Bot。新建项目不会自动加入 Orchestrator，需管理员手动添加。

### 资源监控（性能监控）

**【前端入口】** 左侧「管理」→「性能监控」：展示最近 7 天各 Bot 的任务数、平均耗时。

**API**：`GET /api/tasks/stats?limit_days=7` 返回 `total_tasks`、`per_bot`（username、task_count、avg_latency_ms）。

### 引导 Bot 使用本地 LLM（可选）

@引导 默认根据关键词匹配帮助文档回复。若希望用本地 LLM（如 Ollama）根据文档生成更自然的回答，可在后端目录配置环境变量（或 `.env`）：

| 变量 | 说明 | 示例 |
|------|------|------|
| `GUIDE_LLM_BASE_URL` | OpenAI 兼容 API 根地址 | `http://localhost:11434/v1`（Ollama） |
| `GUIDE_LLM_MODEL` | 模型名 | `qwen2.5:7b` |
| `GUIDE_LLM_API_KEY` | 选填，本地 Ollama 可留空 | 空 |
| `GUIDE_LLM_TEMPERATURE` | 选填，默认 0.7 | `0.7` |
| `GUIDE_LLM_MAX_TOKENS` | 选填，默认 1000 | `1000` |

配置并重启后端后，@引导 的回复将由该 LLM 在「帮助文档」上下文中生成；未配置时仍使用关键词匹配。动态表单（如「帮我创建项目」）在两种模式下都会附带。

### @引导 无反应时如何排查

1. **确认引导 Bot 已存在且已加入当前项目**  
   若从未执行过种子数据，系统里没有「引导」Bot，@引导 不会触发任何回复。请执行一次种子数据：  
   - 启动时加环境变量：`SEED_DATA=1`；或  
   - 手动执行：`cd backend && python -m app.db.seed`（需先 `alembic upgrade head`）。  
   新建项目会自动把引导 Bot 加入该频道（仅当 Bot 账号已存在时）。老项目需在「管理」→「添加成员」中把成员 ID 填为 `bot-guide-001`、类型选 Bot 加入。Orchestrator 的 bot_id 为 `bot-coordinator-001`，需手动加入频道。

2. **确认 @ 的名字是「引导」**  
   消息里必须包含 `@引导`（与 Bot 的 username 完全一致），不能多空格或错字。

3. **若已配置引导 LLM 但希望确认是否连错**  
   查看后端日志：若出现 `guide llm request failed (using keyword fallback)`，说明 LLM 请求失败（地址/端口/模型/网络问题），系统已自动退回关键词匹配；可先修正 LLM 配置或忽略，关键词模式仍会回复。  
   若出现 `no mentioned bots in channel ... mentioned=['引导'] ... channel_bots=[]`，说明当前频道没有「引导」成员，按第 1 步把引导 Bot 加入该项目。

4. **若编排器抛错**  
   日志中会有 `orchestrator failed channel_id=...` 及异常信息，可根据堆栈排查（如数据库、记忆模块等）。

### 日志文件（便于排查错误）

后端会将运行日志写入文件（需配置 `LOG_DIR`，默认 `data/logs`，相对 backend 目录）：

| 文件 | 内容 |
|------|------|
| `agentnexus.log` | 通用日志（INFO 及以上），按大小轮转 |
| `error.log` | 仅错误与异常（ERROR 及以上），便于快速排查 |

未设置 `LOG_DIR` 或目录不可写时，仅输出到控制台。环境变量：`LOG_DIR`、`LOG_MAX_BYTES`（单文件最大字节，默认 5MB）、`LOG_BACKUP_COUNT`（保留份数，默认 3）。未捕获异常会记录完整堆栈到上述日志并返回 500。
