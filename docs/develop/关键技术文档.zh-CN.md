# AgentNexus 关键技术文档

> 来源：概要设计说明书 v2.0 附件一（系统 LLM 使用规范）、附件二（架构决策记录 ADR）

## 1. 系统级 LLM 与任务级 LLM 的区分

| 维度 | 系统级调用（System LLM） | 任务级调用（Task LLM，经 OpenClaw） |
|------|--------------------------|-------------------------------------|
| 发起方 | AgentNexus 内部组件自动触发 | 用户 @mention Bot，由 OpenClaw 发起 |
| 目的 | 维护基础设施（记忆、索引、摘要） | 执行用户交代的专业任务 |
| 用户可见性 | 用户不直接看到调用过程 | 用户直接看到 Bot 回复 |
| 对话历史 | 无多轮，每次独立单次调用 | 携带完整上下文，多轮对话 |
| 输出格式 | 结构化（JSON / Markdown 固定模板） | 自然语言，按任务灵活输出 |
| 模型选择 | 优先最廉价、最快 | 按专业任务选最合适 |
| API Key | 系统统一管理的 System LLM Key | 各 Bot 独立配置的 Task LLM Key |
| 成本承担 | 平台运营成本 | 可按 Bot 或用户分摊 |

## 2. 系统级 LLM 调用场景（5 类）

| 场景 | 触发时机 | 组件 | 输出 | 推荐模型 | 失败处理 |
|------|----------|------|------|----------|----------|
| RECENT.md 滚动摘要压缩 | 每次 Bot 响应后异步 | MemoryManager | 更新后 RECENT，≤1500 字 | Claude Haiku / GPT-4o-mini | 保留旧版，下次重试 |
| 文件摘要（FILES_INDEX 条目） | 文件转 MD 完成后 | FileProcessor → MemoryManager | 3 句话摘要 | Claude Haiku / GPT-4o-mini | 占位符「摘要生成失败」 |
| 图片与 PDF 图片页视觉描述 | 上传图片或 PDF 含图时 | FileProcessor Vision | 图片内容文字描述 MD | Claude Haiku Vision / GPT-4o-mini Vision | 占位符「图片内容无法解析」 |
| 重要决策识别与追加 DECISIONS | Bot 响应后异步判断 | MemoryManager 决策检测 | JSON: is_decision, summary, decision_by | Claude Haiku / GPT-4o-mini | 判断失败则跳过，不写入 |
| 目标对齐偏差检测（可选） | Bot 响应后，管理员开启时 | MemoryManager 目标对齐模块 | JSON: aligned, deviation_score, warning_msg | Claude Haiku | 失败跳过；score>7 告警 |

**成本控制**：系统级调用使用独立 System API Key；推荐廉价模型；RECENT/决策/对齐均为异步，不阻塞主消息流。单频道日均系统 LLM Token 估算约 11.6 万（含可选对齐），成本可忽略。

## 3. 同步与异步原则

- **判断标准**：用户是否需要等该调用完成才能继续？是 → 同步；否 → 异步（进队列，不阻塞）。
- **关键路径（同步）**：Context Store 四层记忆读取、附件 MD 加载（Bot 响应前必须完成）。
- **后台维护（异步）**：RECENT 压缩、文件 3 句话摘要、图片描述、决策识别、目标对齐检测；Bot 回复完成后投递 Redis 队列，Worker 消费，失败不影响对话。

### 文件状态机

- 状态：**上传中** → **转换中** → **已就绪**（或 **转换失败**）。
- Bot 收到含附件消息时：已就绪则加载 MD；转换中则回复「文件仍在处理中，请稍候再提问」；转换失败则回复「该文件无法解析，请确认格式或重新上传」。

## 4. 架构决策记录（ADR）摘要

| 编号 | 问题 | 状态 | 决议摘要 |
|------|------|------|----------|
| D-01 | 是否支持流式输出 | 已落地 | WebSocket 广播和 SSE 消息接口均支持流式输出 |
| D-02 | Context Store 存储方案 | 已演进 | 当前默认 PostgreSQL 存储；MD 文件作为镜像/导出形态。早期 SQLite + MD 双轨方案仅作历史背景 |
| D-03 | 多 Bot 同时被 @ 执行模式 | 已决 | 默认串行；前端展示队列状态与预估时间；暂不设计并行；第二阶段可做「默认协作组」 |
| D-04 | Bot 响应审核策略 | 已决 | 不做全量审核；仅拦截「对外执行类」高风险操作（规则匹配）；第二阶段可升级为 LLM 判断 |
| D-05 | OpenClaw 版本升级兼容 | 已决 | 建立 BotAdapter 隔离层；Orchestrator 只调 Adapter 标准接口；版本升级只改 Adapter，不动 Orchestrator |

### D-04 高风险操作拦截规则（示例）

- 对外发送（如「已发送邮件」）、代码部署（如「已推送到 main」）、文件删除（如「已删除该文件」）、外部 API 调用（如「已完成支付」）→ 拦截并通知管理员确认。
- 建议/分析类（如「建议你发送邮件」）→ 直接发出，不拦截。

### D-05 BotAdapter 接口约定

- `execute(payload: AgentPayload) -> AgentResponse`：唯一对外执行接口。
- `health_check() -> bool`：检查 OpenClaw 实例是否在线。
- 每版本一个 Adapter 子类（如 OpenClawV2Adapter），Orchestrator 不依赖具体版本。

### D-02 当前要点

- 写入：结构化记忆写入数据库；必要时同步到对应 MD 文件作为人工可读镜像。
- 读取：以数据库为准，避免多副本读写冲突。
- 迁移：主业务库与 Context Store 分别通过 `alembic.ini` 和 `alembic_context.ini` 管理。

## 5. 数据存储与扩展

### 5.1 主业务库 PostgreSQL

- 当前默认主业务库使用 PostgreSQL，连接串由 `DATABASE_URL` 提供。
- Alembic 主库迁移使用 `backend/alembic.ini`，Docker backend 启动时默认执行 `upgrade head`。
- 业务数据包括用户、工作空间、频道、成员、Bot、消息、文件元数据等。

### 5.2 Context Store 数据库

- Context Store 连接串由 `CONTEXT_DB_URL` 提供，默认可与主业务库使用同一个 PostgreSQL 数据库。
- Context Store 迁移使用 `backend/alembic_context.ini`，Docker backend 启动时默认执行 `upgrade head`。
- 排查记忆数据时必须以 `.env` 为准：结构化记忆、历史摘要和文件索引来源按 `DATABASE_URL` 查询；旧版 `context_store` 或派生层兼容读写按 `CONTEXT_DB_URL` 查询。

### 5.3 中文消息搜索

- 当前统一搜索入口为 `GET /api/v1/search`，通过 `context`、`types`、`workspace_id`、`channel_id` 限定场景和范围。
- 后续可根据 PostgreSQL 能力引入全文索引或外部检索服务；早期 SQLite FTS5 方案仅作历史参考。

### 5.4 历史 SQLite 迁移说明

- 如果维护者手里仍有早期 SQLite 数据，可采用 [pgloader](https://pgloader.io/) 或导出 SQL/CSV 后通过 PostgreSQL `COPY` 导入。
- 公开版本的新部署不建议再使用 SQLite 作为主库。
