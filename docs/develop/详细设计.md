# AgentNexus 详细设计

> 来源：概要设计说明书 v2.0 §4（核心模块）、§5（易用性）、§6（数据模型）、§7（关键接口）

## 1. 聊天核心（ChatCore）

ChatCore 是 AgentNexus 的聊天基础设施，Python + FastAPI 实现，仅包含本系统所需的最小功能集。

### 1.1 功能范围

| 功能模块 | 说明 | 是否实现 |
|----------|------|----------|
| 频道（Channel） | 创建/管理协作群，公开和私有频道 | ✅ 实现 |
| 实时消息 | WebSocket 广播，Markdown 渲染 | ✅ 实现 |
| @提及（Mention） | @用户 和 @Bot 自动补全与通知 | ✅ 实现 |
| 文件上传与预览 | 上传文件，前端预览 MD 转换结果 | ✅ 实现 |
| Bot Account 管理 | Bot 用户注册、头像、在线状态 | ✅ 实现 |
| 消息历史加载 | 滚动加载历史消息，翻页 | ✅ 实现 |
| 消息搜索 | 全文搜索频道消息和文件内容 | ✅ 实现（基础版） |
| 视频/音频通话、看板、插件市场、企业合规归档 | 超出目标范围 | ❌ 不实现 |

## 2. Agent 编排框架（AgentOrchestrator）

深度参考 CrewAI 的 Crew/Role/Task/Process 概念，并结合聊天场景定制。

### 2.1 核心概念模型

| 概念 | 对应 CrewAI | 在 AgentNexus 中的含义 |
|------|-------------|------------------------|
| Crew（协作组） | Crew | 一个 Channel 内被激活的所有 Bot 集合 |
| Agent（智能体） | Agent | 每个 OpenClaw 实例，具备 Role、Goal、Backstory |
| Task（任务） | Task | 用户的一次 @提及触发，含输入消息、期望输出、上下文 |
| Process（执行模式） | Process | Sequential（顺序）或 Parallel（并行），多 Bot 被 @ 时的响应方式 |
| Coordinator（协调者） | Manager Agent | 可选主控 Bot，任务拆解与结果汇总（第三阶段） |

### 2.2 消息路由逻辑（第一阶段：显式 @mention）

- 解析消息中的 @BotName，得到被 @ 的 Bot 列表。
- 与本频道已激活的 Bot 取交集，得到目标 Bot。
- 从 MemoryManager 加载四层记忆，从 FileProcessor 加载附件 MD。
- 默认**串行**执行：按 @ 顺序依次调用各 Bot，前一个完成后再调用下一个（见附件二 D-03）。
- Orchestrator 仅在**同一条任务线上串行选择 Bot**，不拆分子任务、不设计 Tool 流程，也不在平台层进行多步规划。

### 2.2.1 顶层执行流与 task_id

- 一次「用户顶层提问 + 多个 Bot 串行协作」在 AgentNexus 视角是**单线流式链条**，而不是树状分叉。
- 对这条链分配统一的 `task_id`，该 `task_id` 贯穿：
  - 用户顶层提问消息；
  - 每一次对 OpenClaw 的调用（AgentPayload.task_id）；
  - 所有 Bot 的回复与建议（写入 AgentTask 与消息流）。
- `task_id` 被视作**任务线程 id**，主要用于：
  - 任务监控、性能统计、错误排查；
  - 在线上以「线程视图」还原从用户消息、到 Orchestrator 决策、再到各 Bot 执行的完整链路。

### 2.3 Bot 角色配置（SOUL.md）

每个 OpenClaw 实例通过 SOUL.md 定义专业角色，AgentNexus 注册 Bot 时读取。示例：@codebot（代码审查）、@docbot（技术文档）、@databot（数据分析）、@planbot（项目规划）、@searchbot（知识检索）、@coordinator（Orchestrator/主控，门户阶段一已实现）。

### 2.4 Orchestrator 与内置 Bot（已实现）

| 能力 | 说明 |
|------|------|
| **直接回答** | 频道开启 `auto_assist` 或全局开启直接回答后，用户未 @ 时由内置 channel bot 接管 |
| **显式 @channel bot** | 内置 channel bot 负责引导、澄清、路由建议与聚合回答 |
| **建议 @部门bot** | Orchestrator 回复可含「建议 @xxx 回答 XXX」；orchestrator_auto_takeover 开启时被建议 Bot 自动接手 |
| **自动接手 UI** | WebSocket 推送 bot_processing 事件，前端显示「@xxx 正在处理...」 |

内置执行核心为统一 `channel bot`（`bot-guide-001`）；默认 Bot 执行路径为 `LLMBotAdapter`，OpenClaw HTTP/WS 适配器作为可选接入能力保留。

## 3. 四层记忆体系（MemoryManager）

为每个协作频道建立四层独立记忆结构；每次调用 OpenClaw 前，MemoryManager 将四层记忆拼接为 System Prompt 前缀注入，使不同 Bot 通过共享 Context Store 获得一致项目视角。

| 记忆层 | 文件/存储 | 内容 | 更新方式 | 大小上限 |
|--------|-----------|------|----------|----------|
| 第一层：项目锚点 | ANCHOR.md | 核心目标、关键约定、成员职责、重要限制 | 管理员维护或 Bot 生成摘要 | 约 1500 字 |
| 第二层：决策记录 | DECISIONS.md | 重要决策（时间戳、决策人、依据、结论） | Bot 自动追加，人工可编辑 | 约 3000 字 |
| 第三层：资料索引 | FILES_INDEX.md | 已上传文件摘要索引（文件名 + 3 句话摘要 + 上传人） | 文件处理完成后自动写入 | 无硬上限 |
| 第四层：近期动态 | RECENT.md | 最近 50 条消息的压缩摘要 | 每次 Bot 交互后轻量 LLM 压缩更新 | 固定约 1500 字 |

**上下文注入格式**（拼接后作为 System Prompt 前缀）：

- 项目锚点（最高优先级）→ 重要决策记录 → 已上传资料索引 → 近期频道动态。
- 本地排查 PostgreSQL 记忆数据时，必须先读取 `.env`：`memory_entries`、`history_pages`、`file_records` 等主库表按 `DATABASE_URL` 查询；旧版 `context_store` 或派生层兼容读写按 `CONTEXT_DB_URL` 查询，不要直接套用示例连接信息。

## 4. 文件处理与存储（FileProcessor + Storage）

### 4.1 文件上传主流程（已实现）

- 主路径：`POST /api/files/presign` 返回预签名上传参数，前端直传对象存储
- 兼容路径：`POST /api/files/upload`（legacy）
- 状态查询：`GET /api/files/{file_id}/status`
- 预览/下载：`GET /api/files/{file_id}/preview`、`GET /api/files/{file_id}/download`

### 4.2 存储抽象（已实现）

- 接口：`StorageProvider`（`app/storage/base.py`）
- 启动初始化：`app/storage/bootstrap.py`
- S3 兼容实现：`app/storage/s3_compatible.py`

| 格式 | 处理工具 | 说明 |
|------|----------|------|
| .txt / .md | 直接使用 | 无需转换 |
| .docx | mammoth | 保留标题、列表、表格 |
| .pdf | pymupdf（fitz） | 文字与表格；图片页 Vision API 描述 |
| .xlsx / .csv | openpyxl / pandas | 转 Markdown 表格；大表可截取摘要 |
| .png / .jpg | Claude Vision / GPT-4o Vision | 生成内容描述 MD |
| 超长文档（>3 万字） | 分块 + 索引策略 | 按需启用，避免上下文过载 |

## 5. 易用性设计（摘要）

- **前端**：零学习成本、@ 提及引导、Bot 状态可见、响应结果带角色标签、文件上传进度、上下文摘要面板、一键拉入 Bot、记忆编辑入口。
- **管理后台**：Bot 添加向导（选类型→配 LLM→设角色→测试→分配频道）、系统状态监控（仪表盘、统计卡片、友好错误、一键重启）、频道上下文管理（ANCHOR/DECISIONS/FILES_INDEX 图形化、重置记忆）。

## 6. 数据模型

### 6.1 核心实体

| 实体 | 关键字段 | 说明 |
|------|----------|------|
| Workspace | workspace_id, name, created_at | 顶层组织单元 |
| Channel | channel_id, workspace_id, name, type(public/private), purpose, auto_assist | 核心协作单元（支持频道级自动接管） |
| User | user_id, username, display_name, role(admin/member/botmanager), avatar_url | 人类用户 |
| BotAccount | bot_id, username, display_name, description, avatar_url, model_id, template_id, custom_system_prompt, status | Bot = 模型 + 模板，可选自定义 system prompt |
| WorkspaceMembership | workspace_id, user_id, role(owner/admin/member), joined_at | 工作空间成员关系 |
| ChannelMembership | channel_id, member_id, member_type(user/bot), joined_at, added_by | 频道成员关系 |
| Message | msg_id, channel_id, sender_id, sender_type, content, file_ids[], mention_bot_ids[], **task_id**, **in_reply_to_msg_id**, created_at | 统一消息存储，支持任务线程与问答配对 |
| ContextStore | channel_id, layer(ANCHOR/DECISIONS/FILES_INDEX/RECENT), content, updated_at, updated_by | 四层记忆缓存（附件二为 SQLite + MD） |
| FileRecord | file_id, channel_id, uploader_id, object_key, storage_provider, status, summary_3lines, converted_at | 文件处理与存储状态 |
| AgentTask | task_id, channel_id, bot_id, trigger_msg_id, response_msg_id, latency_ms, token_count, feedback | 任务日志与质量监控 |

> 本地 PostgreSQL 查询提醒：所有记忆相关表的实际连接信息以 `.env` 为准；主业务表走 `DATABASE_URL`，旧版 Context Store 走 `CONTEXT_DB_URL`。

#### 7.2.1 问题–回答配对与线程标识

- **`task_id`（线程 id）**
  - 由 Orchestrator 在收到一次可触发 Bot 的顶层消息时生成；
  - 在同一条任务线上保持不变，贯穿所有后续 AgentPayload / AgentResponse 与 AgentTask 记录；
  - 用于将「A Bot 建议 @B」「Orchestrator 决策」「B Bot 执行」等行为串联为一条可观测的执行流。
- **`in_reply_to_msg_id`（问答精确指针）**
  - Message 表中新增字段，用于指向“本条消息回复的是哪一条消息”；
  - 前端构建问答卡片与折叠展示时，**主要依赖 `in_reply_to_msg_id`** 来确定每条回答归属到哪个问题；
  - Orchestrator 在写入 Bot 回复消息时，统一将当前 AgentPayload.trigger_message.msg_id 写入 `in_reply_to_msg_id`。
- 二者分工：
  - UI 问答折叠：依赖 `in_reply_to_msg_id` 确保每条回答都知道自己的问题是谁；
  - 任务监控与日志：依赖 `task_id` 在 Admin 页面和日志系统中展示整条任务线程的演进。

### 7.3 SSE 事件流约定（已实现）

`POST /api/channels/{channel_id}/messages/stream` 使用 `text/event-stream`，核心事件：

- `user_message`：用户消息已落库
- `bot_processing`：某个 bot 开始处理
- `delta`：增量 token
- `done`：单个 bot 流结束
- `complete`：整次请求结束
- `error`：处理异常
