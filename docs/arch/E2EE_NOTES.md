## 核心张力（务必先理解）

AI Agent 聊天平台里**不存在单一的 E2EE**：
**bot 要回复就必须读明文，所以有 bot 的会话里 bot 必须成为一个持密钥的"端"。**
这与 Signal 式"只有端点持钥、服务端只见密文"有根本冲突。

依赖明文的服务端功能（E2EE 下全部失效或须挪到客户端）：
- 全文搜索 `search_service`
- RAG / embedding `memory/files_index`
- 历史摘要 `channel_memory`
- 文件转换 `file_processor`

---

## 好消息：本架构很多部件天生兼容

| 部件 | 兼容性 |
|------|--------|
| Rust Backend 实时层 | ✅ 哑管道，不解析 `data`，密文原样转发，零改动 |
| PostgreSQL / S3（无 NATS） | ✅ 存转密文即可 |
| `msg_id` 回流关联 | ✅ msg_id 是元数据非内容，机制存活 |
| `seq` 去重 | ✅ 元数据，存活 |
| 服务端**构造**消息内容 | ❌ 服务端无法生成密文，须改由端点加密 |

> 即：连接 / 关联 / 顺序机制全部存活，只是 `data` 变成不透明密文。

---

## 关键洞察：ACP connector 是天然的链下加密端点

- **内部 LLM bot**：worker 要调 LLM 就得解明文 → worker 在服务端 → **对运营方非 E2EE**（除非 TEE/机密计算，巨大工程，且 LLM 提供商仍见明文）。
- **外部 ACP bot**：connector 跑在用户/合作方机器上，在服务端信任边界**之外**。bot 私钥只放 connector → **服务端真读不到明文**，本地 agent 解密处理 → **唯一能做到"带 bot 的真 E2EE"的路径**，且与现有 ACP 架构契合。

## 三个自洽层级（按威胁模型选）

| 层级 | 防护对象 | bot 可用 | 搜索/RAG | 工程量 |
|------|---------|---------|----------|--------|
| A. 传输+静态加密（TLS+落盘） | 网络窃听、磁盘失窃 | ✅ 全部 | ✅ 正常 | 小（基本已有） |
| B. 人-人会话 E2EE（无 bot 房间） | 含运营方 | ❌ 该房禁 bot | ❌ 该房禁用 | 中 |
| C. ACP 端点 E2EE（私钥在 connector） | 运营方 | ✅ 仅 ACP bot | ❌ 挪 connector 端 | 大 |

---

## E2EE 真上要付的代价

- 搜索/RAG/摘要 → 客户端或 connector 端做，或 E2EE 房间禁用
- 文件 → 客户端加密后上传，S3 存密文；大小/类型元数据仍泄露（除非填充）
- 推送 → 不能带明文预览
- 密钥管理 → 设备身份密钥 + 群组 sender-key/逐成员包裹 + 公钥目录（TOFU 或带外验证/透明日志防 MITM）
- **元数据**（谁跟谁/何时/成员/时序/大小）**仍对服务端可见**；sealed sender 等元数据保护通常超范围

## 待调研问题（将来接续时回答）

1. 威胁模型到底防谁？（网络 / 磁盘 / 运营方 / 跨租户）——决定 A 还是 B/C。
2. 哪些会话要 E2EE？人-人 only，还是含 ACP bot 房间？——决定 bot 能否参与。
3. 群组密钥方案：sender-key（Signal 群组）vs 逐成员逐消息包裹？
4. 公钥目录信任：服务端目录 + 带外验证？还是引入 key transparency？
5. 搜索/RAG 的客户端化成本是否可接受？否则 E2EE 房间直接禁用这些功能。
6. 流式场景：ACP connector 对每个 delta 加密到频道成员的开销 / 批量策略。

---

## 现在的取舍

- **本次重构按层级 A 推进**（TLS + 静态加密），不引入 E2EE。
- 设计上**保留兼容性**：Backend 实时层哑管道、msg_id/seq 元数据关联——这些不阻碍将来上 B/C。
- B/C 留作未来计划，调研先回答上面 6 问。
