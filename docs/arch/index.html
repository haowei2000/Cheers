关键运行流

下面是当前主链路的核心契约：写后投递、消息级 bot 路由、Agent Bridge 生命周期，以及 ACP MCP stdio 的责任边界。

1 消息写入 验证成员、解析 token mention、事务分配 channel_seq、插入消息。
2 终态 fan-out PG commit 后再广播浏览器终态帧，满足 Write-Before-Deliver。
3 message-level dispatch 有 @bot 派发指定 bot；bot 回复再次 mention bot 时以 depth+1 继续派发，超过上限静默。
4 per-message cancel 用户停止某条 bot placeholder 时，Backend 找到 owner bot 并只推送该消息的 cancel frame。
5 connector MCP 注入 connector 在 ACP session/new/load 注入 mcpServers；ACP agent 启动 stdio MCP 子进程，MCP tool call 回到 connector loopback。
6 资源裁判 connector 把 MCP tool call 转成 resource_req，Backend 统一校验频道成员、Grant 和 capability delegation。

已经明确的架构方向

这些决策在当前代码或文档中已经形成约束，后续实现应继续沿用。

外部 agent 优先

平台不再依赖内置 Python bot runtime；外置 ACP bot 和 MCP agent 是同一协议路径。

频道只转发 @

没有中央 coordinator 参与意图判断，路由规则可测试、可审计。

资源写走 Grant

Resource API 读只需频道成员，写需要成员身份和 ACP RBAC Grant。

MCP 注入归 connector

daemon 只管理 connector 生命周期；connector 注入 mcpServers，ACP agent 按 stdio 启动 MCP server。

Agent Bridge 是平台协议

服务端不直接说 ACP JSON-RPC；ACP 只在 connector runtime adapter 层出现。

单实例是本期约束

fan-out 和 BotLocator 先走进程内或 Redis registry，跨实例扩展保留 trait 接口。

旧兼容只保留在边界

OpenClaw 和独立 TS bridge-client 退场；TS foreground runtime 只是 Rust connector 全量化前的兼容层。
