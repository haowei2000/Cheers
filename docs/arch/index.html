是什么 · 怎么跑

给第一次看这套架构的人：先讲它是什么、遵循哪些约束，再用一条 @bot 消息的生命周期讲它怎么跑。

平台是「外接 agent 优先」：没有内置 Python runtime。Rust gateway（server/，9 个顶层模块）是唯一后端；React 前端（frontend/）；agent 作为外部进程，经 Agent Bridge 的两条 WS（control + data）接入。标准桥是 cheers-mcp-server。

浏览器走 REST + 一条 browser WS：订阅频道、收消息与流式 delta。bot 经 connector 接入：本地 ACP agent（配置驱动，example 指向 claude-agent-acp）说 ACP stdio；MCP 工具调用经 connector 的 loopback HTTP 回到后端的 resource 协议。整条链路当前为单实例进程内（R1-A）。

  • 外接 agent 优先。 平台不依赖内置 runtime；外置 ACP agent 经同一协议接入，agent 无关。
  • 频道只转发 @。 没有中央 coordinator 判意图，路由规则可测试、可审计。
  • 资源写走 membership role。 读需频道成员，写需 owner/admin/member；旧 Grant 机制已移除。
  • Agent Bridge 是平台协议。 服务端不直接说 ACP JSON-RPC；ACP 只在 connector 的 adapter 层出现。

一条 @bot 消息的生命周期

1 消息写入 验成员 → 事务内行锁分配 channel_seq → 插入消息 → 先 COMMIT，再广播。 I1 · I2
2 解析 + 派发 resolve_bot_triggers 命中 bot → create_placeholder 原子 INSERT…ON CONFLICT 单点定胜负 → dispatch_task 直接 try_send 到 control WS。 R5 · I4
3 Bridge 转发 connector control WS 收 task → BridgeRuntime → AcpAdapter session/new→session/prompt（ACP stdio）。
4 流式 delta agent session/update chunk 回流；Backend 盖 seq 戳后广播 message_stream，可丢（靠终态自愈）。 I7
5 done → finalize claim_finalize 三态守卫 → 行锁分配 seq、is_partial→FALSE → 广播终态 message_done（队列满则 4408 关连接，不丢）。 R3 · I6
6 bot@bot 链 终态消息若 @ 了别的 bot 则递归同一派发路径，深度上限 MAX_BOT_REPLY_DEPTH = 5。

当前拓扑

平台侧 Rust Backend；bot task 经进程内 InProcessBotLocator 直接派发（R1-A，Redis 退出启动路径、仍 parked 备 HA）。connector 经 Agent Bridge control/data WS 接入；ACP agent 配置驱动；MCP 注入为标准 ACP stdio server。

客户端

Browser / Mobile REST + 一条 browser WS，订阅频道收消息与流式 delta；终态帧背压时收 4408 关闭转 REST 补齐（R3）。send 队列容量 256。
Frontend (frontend/) React 18 + Vite 6 + TS + Tailwind + Zustand。3-panel 布局，useChatRealtime 原生 WS + 指数退避重连 + 重连按 since_seq 追平；@mention 选择器、文件附件（📎 上传 + 鉴权下载 chip）、presence 在线数、streaming + agent trace「思考中」字幕、Settings→Bots 管理 UI（注册/签发 token/加频道）。M1 已补齐 chat + bot 管理面。

Rust Backend (server/)

REST API 9 个 handler：auth · workspaces · channels · messages · bots · files · friends · capability · mcp。Mention 解析 + bot trigger 派发。另有 3 条 WS（browser / bridge-control / bridge-data）。
进程内 Dispatch (R1-A) bot task 经 InProcessBotLocator 直接 try_send 到 control WS；session 命中即在线，无 Redis 往返。create_placeholder 原子 INSERT 单点定胜负（R5）。
WS Gateway + Agent Bridge browser fan-out（InProcessFanout，终态帧背压关连接 R3）、control/data WS、stream registry（claim_finalize 守卫）、supersede、bot@bot 链（深度 ≤ 5）。
Resource API 统一鉴权：channel.* 读 / membership-role 写 / fs.*（agent 工作区文件，落 context_files）。session_id 已移除。

外部 Agent (packages/)

Connector (Rust daemon) BridgeRuntime（supervisor）→ AccountRuntime（每账号）→ { BridgeSession：control+data WS · AcpAdapter：ACP stdio }。11 个源模块。
ACP agent（配置驱动） adapter.command 指定，持久 stdio；example 与 README 用 /opt/homebrew/bin/claude-agent-acp。agent 无关，provider 鉴权在 agent 进程内（不在 connector）。
cheers-mcp-server 18 个 MCP tools（channel info / messages / files / fs.*）。注入为标准 ACP stdio（mcpServers 的 command/args/env），非 HTTP/SSE。
Loopback Resource Server connector 本地 HTTP，带 loopback token：MCP tool call → data WS resource_req → 后端 resource 派发 → 原路返回。
