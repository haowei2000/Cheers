# Bot Config Governance — sovereignty layers (connector/ACP era)

> Status: design for the **current** architecture (Rust gateway + external ACP
> connector). This is the **sovereignty axis** — *who may change what, and where
> it lives*. It is orthogonal to (and supersedes, for the connector world) the
> older [BOT_CONFIG_LAYERING.md](./BOT_CONFIG_LAYERING.md), which describes a
> *scope/inheritance* axis (library → bot → channel merge) from the retired
> Python Agent Service. Reuse that doc's merge rules (limits=min, tools=∩,
> security only-tightens); ignore its Python placement.

## 0. Governing principle

**The connector TOML is the host's capability envelope. The platform (Cheers) is
read-only against it.** The TOML not only holds host wiring/secrets/sandbox — it
also declares, via `backend_may_*` gates, *what the platform is allowed to change
at runtime*.

Corollary (the user's rule): **anything the platform must change does NOT live in
the TOML as a value; the TOML holds only the boundary (gate) + a default/ceiling.**

## 1. The three layers

| Layer | Owner | What | Where it lives |
|---|---|---|---|
| **L0 — Host-sovereign (TOML)** | Host op; platform read-only | (a) wiring/trust/sandbox: `bridge.*`, `adapter.command/args`, `policy.env`, `policy.workspace.allowed_roots`, `policy.mcp.allowed_servers`, send/prompt/session caps · (b) **capability gates + defaults/ceilings**: `policy.workspace.backend_may_set_cwd`, `policy.config.backend_may_set_model`, `policy.mcp.backend_may_inject_extra_servers`, `policy.permission.forward_to_backend`/`auto_allow`, the default `permission_mode`, and the set-mode envelope (`backend_may_set_mode` + opaque `allowed_modes`) | `~/.cheers/*.toml` (generated by `server/src/domain/connector_config.rs`) |
| **L1 — Bot config (persisted, platform)** | Platform | (a) Cheers-native: `display_name`, `avatar`, `intro`, `scope`, `template_id`, `custom_system_prompt`, `model_id`, `is_disabled`, triggers · (b) ACP **config-options** desired-state: agent-declared `configOptions` (model, effort, …) | `bot_accounts` (+ `binding_config.connector_control` snapshot); applied via `session/set_config_option` |
| **L2 — Runtime session controls (imperative)** | Platform issues, L0 validates | `session/set_mode` (permission posture), set cwd, runtime set model; per live session, re-applied on each new session from L1 | ACP `session/set_mode` / `session/set_config_option` (connector `acp_runtime.rs` / `acp_adapter.rs`) |

## 2. The invariant (one chain per knob)

> **L0 gate + default → L1 persisted desired-value → L2 applied to the session (clamped by L0).**

Every platform-controllable runtime setting follows this. The platform may never
exceed the L0 envelope (can't set cwd outside `allowed_roots`, can't pick a mode
the gate forbids, can't enable bypass if `allow_bypass=false`).

## 3. Case studies

- ✅ **model — the pattern to copy.** L0 gate `backend_may_set_model`; L1 `model_id`; L2 `session/set_config_option`. Platform-settable, host-vetoable.
- ❌ **permission_mode — the anti-pattern (current bug).** Hardcoded in TOML
  (`adapter.permission_mode`), so the platform can't change it → it's both
  mis-placed (a platform-changeable value living in L0) **and** mis-valued
  (`"plan"` = "no execution"; the mode that prompts per-tool is `"default"`).
  Fix: L0 holds only the *default + allowed set*; the live value is L1-persisted
  and L2-applied via `session/set_mode`.
- 🆕 **set-mode envelope — new L0 gate (ACP-generic).** `backend_may_set_mode`
  (may the platform change mode at all) + `allowed_modes` (an **opaque allow-list
  of ACP modeIds**; empty = any mode the agent advertised in `session/new`). The
  connector matches by exact string and has **no notion of what a mode means** —
  so it stays agent-agnostic. Agent-specific knowledge (e.g. *which* mode is
  "bypass") lives **only in the gateway's per-agent preset**: e.g. the claude
  preset lists its safe modes and simply **omits `bypassPermissions`**, which is
  how "no bypass by default" is expressed — without any non-ACP `allow_bypass`
  concept leaking into generic code. Unknown agents → empty list (any advertised
  mode); the host may restrict in their TOML.

## 4. Open item (needs a decision)

`claude-agent-acp` launches the underlying Claude SDK with
`--allow-dangerously-skip-permissions` whenever `ALLOW_BYPASS` is true, and
`ALLOW_BYPASS = !IS_ROOT || IS_SANDBOX` (acp-agent.js) — i.e. **always true for a
normal non-root host**. That flag is *below* the ACP `session/set_mode` layer, so
an L0 `allow_bypass=false` cannot be honored purely from our connector config; it
requires either patching/forking `claude-agent-acp` (don't pass the flag unless
our gate allows it) or an upstream change. **Until that's resolved, L2 `set_mode`
controls posture but the bypass flag's exact effect on the per-tool approval card
is unverified.** Tracked as the blocker for the "approval cards actually fire" goal.

## 5. Implementation order

1. Fix the `connector_config.rs` preset: `permission_mode` default `plan` → `default` (+ correct the comment).
2. Add the L0 set-mode envelope: connector `backend_may_set_mode` + opaque `allowed_modes` (string-match clamp, no semantics); agent-specific mode lists only in the gateway's per-agent preset (claude omits its bypass mode).
3. L1: persist per-bot `permission_mode` (+ allowed set) and `allow_bypass` (bot_accounts / binding_config); expose in the bot config API + UI.
4. L2: connector applies the L1 value via `session/set_mode`, clamped to the L0 allowed set.
5. Resolve §4 (bypass flag) so `allow_bypass=false` truly disables bypass; then verify approval cards fire end-to-end.
