# Cheers Admin Guide

> **Language**: English | [中文](系统管理说明书.zh-CN.md)

This guide is for administrators who create workspaces and channels, manage members, create Bots, and connect local ACP-capable agents through Agent Bridge. OpenClaw links are legacy/deprecated; prefer ACP local agents for new deployments.

## Admin Entry Points

| Entry | URL |
|---|---|
| Frontend | Docker: `http://localhost`; Vite dev: `http://localhost:5173` |
| Health check | `http://localhost:8000/health` |
| Connector discovery (authenticated) | `GET http://localhost:8000/api/v1/ops/connector-discovery` — reports the control/data WebSocket URLs an ACP connector should dial |

Administrators and users share the same frontend. Common operations are available in the Manage area; advanced operations can be performed through the API. The Rust gateway does not serve a Swagger UI — the REST surface is defined in `server/src/router.rs`.

## DingTalk Login Administration

When DingTalk login is enabled, users can sign in from the login modal with **Continue with DingTalk**. Browser users are sent through DingTalk OAuth; users inside the DingTalk client use the in-app delegated authorization flow when a default or single allowed `corpId` is configured.

Security defaults:

- DingTalk login is off unless `DINGTALK_LOGIN_ENABLED=true`.
- `DINGTALK_ALLOWED_CORP_IDS` is required; users from other DingTalk organizations are rejected.
- First-time DingTalk users are auto-created with the Cheers `member` role only.
- Cheers does not store DingTalk access or refresh tokens.
- Existing password accounts are not linked by email or mobile number automatically.

## Concepts

| Concept | Description |
|---|---|
| Workspace | Top-level organization container. Channels belong to a workspace. |
| Project / Channel | Collaboration room where people and Bots chat. |
| Member | A `user` or `bot` added to a channel. |
| Bot account | A Bot identity that can be mentioned in channels. Backed by an external ACP agent (via Agent Bridge) or an OpenAI-compatible model — there is no built-in bot. |

## Create a Workspace

Recommended: use the frontend Manage area.

API example:

```bash
curl -X POST http://localhost:8000/api/v1/workspaces \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{"name":"Default Workspace"}'
```

Record the returned `workspace_id`.

## Create a Channel

```bash
curl -X POST http://localhost:8000/api/v1/channels \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{"workspace_id":"ws-default-001","name":"My First Project","type":"public","purpose":"Daily collaboration"}'
```

The response contains `channel_id`, which is required for member operations.

## Manage Members

List channel members:

```bash
curl "http://localhost:8000/api/v1/channels/<channel_id>/members" \
  -H "Authorization: Bearer <access_token>"
```

Add a user or Bot:

```bash
curl -X POST "http://localhost:8000/api/v1/channels/<channel_id>/members" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{"member_id":"<user_or_bot_id>","member_type":"bot"}'
```

Remove a member:

```bash
curl -X DELETE "http://localhost:8000/api/v1/channels/<channel_id>/members/<member_id>" \
  -H "Authorization: Bearer <access_token>"
```

Users can also invite Bots from the chat composer by typing `@` and selecting a Bot that has not joined the channel yet.

## Bot Binding Types

| `binding_type` | Adapter | Description |
|---|---|---|
| `http` | HTTP Bot | Calls an OpenAI-compatible model using the Bot's AIModel and PromptTemplate. |
| `agent_bridge` | Agent Bridge Bot | External provider connects back through `/ws/agent-bridge/control` and `/ws/agent-bridge/data`. Prefer ACP Connector; OpenClaw is legacy/deprecated. |

There is no built-in bot — the old `Coordinator` (`bot-helper-001`) was removed with the Python backend. Every Bot is created by an administrator or user and backed by an external agent or model.

## Register an Agent Bridge Bot

Recommended frontend flow:

1. Open Manage -> Bot Management.
2. Create a Bot and choose **Agent Bridge Bot**.
3. Fill in username, display name, provider agent id, and optional prompt/template settings.
4. Save the one-time `agb_...` token immediately.
5. Add the Bot to the target channel.

For scripted onboarding of the host that runs the agent, the gateway exposes a one-time **enrollment code** flow (all paths verified against `server/src/router.rs`):

| Step | Endpoint | Notes |
|---|---|---|
| Mint an enrollment code | `POST /api/v1/bots/{bot_id}/enrollment` | Owner/admin JWT. Plaintext code returned **once**; single-use, 15-minute TTL. `DELETE` on the same path revokes all live codes for the bot. |
| Redeem the code on the agent host | `POST /api/v1/enrollment/redeem` | Public — authenticated by the code itself. Atomically claims the code, rotates the bot token, and returns the token plus a ready-to-run connector config. |
| One-line connector installer | `GET /api/v1/install.sh` | Public. Serves an installer with the API base baked in; it reads the code from `CHEERS_ENROLL_CODE` and redeems it itself. |
| Install guidance | `GET /api/v1/enrollment/guidance` | Authenticated. Returns the install URL and a prompt template you can hand to an agent. |
| Connector config (manual mode) | `GET /api/v1/bots/{bot_id}/connector-config` | Owner/admin. Ready-to-run config that reads the token from a sidecar file; issue the token separately via `POST /api/v1/bots/{bot_id}/token`. |

The old unauthenticated `/docs/agent-bridge/*` endpoints (`register`, `help`, `discovery`, `auth-check`) belonged to the removed Python backend and no longer exist.

## Connector Discovery

A connector (or the onboarding wizard) can ask the gateway where to dial:

```bash
GET /api/v1/ops/connector-discovery   # authenticated
```

The response contains the public WebSocket base (`CHEERS_CONNECTOR_PUBLIC_BASE` if configured, otherwise a localhost fallback) and the resolved `control_url` / `data_url` for the two Agent Bridge WebSockets, `/ws/agent-bridge/control` and `/ws/agent-bridge/data`.

## Closing Agent Bridge Sessions

Bot administrators can close stale Agent Bridge sessions from the Bot Status panel. Closing marks the session `closed`, detaches active scope bindings, and keeps the historical record for auditing and troubleshooting. The same action is available through:

```bash
DELETE /api/v1/channels/{channel_id}/bots/{bot_id}/sessions/{session_id}
```

## A Bot Does Not Reply

1. Confirm the Bot exists, is enabled, and is a member of the current channel.
2. Confirm the message contains the Bot's exact `@username` (routing is a deterministic `@mention → bot` lookup; a message that mentions nobody triggers no bot).
3. For Agent Bridge Bots, confirm the provider/connector is connected — check the Bot Status panel or `GET /api/v1/bots/{bot_id}/status`.
4. Check gateway logs for Bot runtime or provider errors.
