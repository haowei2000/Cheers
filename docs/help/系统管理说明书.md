# Cheers Admin Guide

> **Language**: English | [中文](系统管理说明书.zh-CN.md)

This guide is for administrators who create workspaces and channels, manage members, create Bots, and connect local ACP-capable agents through Agent Bridge. OpenClaw links are legacy/deprecated; prefer ACP local agents for new deployments.

## Admin Entry Points

| Entry | URL |
|---|---|
| Frontend | Docker: `http://localhost`; Vite dev: `http://localhost:5173` |
| Swagger | `http://localhost:8000/docs` |
| Health check | `http://localhost:8000/health` |
| Agent Bridge discovery | `http://localhost:8000/docs/agent-bridge/discovery` |

Administrators and users share the same frontend. Common operations are available in the Manage area; advanced operations can be performed through the API.

## DingTalk Login Administration

When DingTalk login is enabled, users can sign in from the login modal with **Continue with DingTalk**. Browser users are sent through DingTalk OAuth; users inside the DingTalk client use the in-app delegated authorization flow when a default or single allowed `corpId` is configured.

Security defaults:

- DingTalk login is off unless `DINGTALK_LOGIN_ENABLED=true`.
- `DINGTALK_ALLOWED_CORP_IDS` is required; users from other DingTalk organizations are rejected.
- First-time DingTalk users are auto-created with the Cheers `member` role only.
- Cheers does not store DingTalk access or refresh tokens.
- Existing password accounts are not linked by email or mobile number automatically.

## Concepts

| Concept | Description |
|---|---|
| Workspace | Top-level organization container. Channels belong to a workspace. |
| Project / Channel | Collaboration room where people and Bots chat. |
| Member | A `user` or `bot` added to a channel. |
| Bot account | A Bot identity that can be mentioned in channels. |
| Coordinator | Built-in assistant with `bot_id=bot-helper-001` and username `Coordinator`. |

## Create a Workspace

Recommended: use the frontend Manage area.

API example:

```bash
curl -X POST http://localhost:8000/api/v1/workspaces \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{"name":"Default Workspace"}'
```

Record the returned `workspace_id`.

## Create a Channel

```bash
curl -X POST http://localhost:8000/api/v1/channels \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{"workspace_id":"ws-default-001","name":"My First Project","type":"public","purpose":"Daily collaboration"}'
```

The response contains `channel_id`, which is required for member operations.

## Manage Members

List channel members:

```bash
curl "http://localhost:8000/api/v1/channels/<channel_id>/members" \
  -H "Authorization: Bearer <access_token>"
```

Add a user or Bot:

```bash
curl -X POST "http://localhost:8000/api/v1/channels/<channel_id>/members" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <access_token>" \
  -d '{"member_id":"<user_or_bot_id>","member_type":"bot"}'
```

Remove a member:

```bash
curl -X DELETE "http://localhost:8000/api/v1/channels/<channel_id>/members/<member_id>" \
  -H "Authorization: Bearer <access_token>"
```

Users can also invite Bots from the chat composer by typing `@` and selecting a Bot that has not joined the channel yet.

## Bot Binding Types

| `binding_type` | Adapter | Description |
|---|---|---|
| `http` | HTTP Bot | Calls an OpenAI-compatible model using the Bot's AIModel and PromptTemplate. |
| `agent_bridge` | Agent Bridge Bot | External provider connects back through `/ws/agent-bridge/control` and `/ws/agent-bridge/data`. Prefer ACP Connector; OpenClaw is legacy/deprecated. |

The built-in Coordinator uses the fixed `bot_id=bot-helper-001`. Historical names are only compatibility aliases; new docs and onboarding should use `@Coordinator`.

## Register an Agent Bridge Bot

Recommended frontend flow:

1. Open Manage -> Bot Management.
2. Create a Bot and choose **Agent Bridge Bot**.
3. Fill in username, display name, provider agent id, and optional prompt/template settings.
4. Save the one-time `agb_...` token immediately.
5. Add the Bot to the target channel.

Machine-readable registration is also available:

```bash
POST /docs/agent-bridge/register
```

The response returns `bot_token`, `controlUrl`, `dataUrl`, and a recommended Rust ACP Connector TOML snippet. The plaintext token is shown only once.

## Agent Bridge Discovery

External providers should start with:

```bash
GET /docs/agent-bridge/discovery
```

The response contains registration URLs, auth information, WebSocket endpoints, file APIs, and optional plugin download metadata.

OpenClaw plugin metadata in the discovery response is legacy/deprecated. Keep it only for existing OpenClaw deployments, and migrate new or actively maintained integrations to `/acp-bridge` with the Rust ACP connector and a local ACP-capable agent.

## Closing Agent Bridge Sessions

Bot administrators can close stale Agent Bridge sessions from the Bot Status panel. Closing marks the session `closed`, detaches active scope bindings, and keeps the historical record for auditing and troubleshooting. The same action is available through:

```bash
DELETE /api/v1/bots/{bot_id}/sessions/{session_id}
```

## Coordinator Local LLM Configuration

Coordinator can use local or OpenAI-compatible LLM configuration through the historical `HELPER_LLM_*` variables:

| Variable | Meaning |
|---|---|
| `HELPER_LLM_BASE_URL` | OpenAI-compatible base URL, for example `http://localhost:11434/v1` |
| `HELPER_LLM_MODEL` | Model name |
| `HELPER_LLM_API_KEY` | Optional API key |
| `HELPER_LLM_TEMPERATURE` | Optional temperature, default `0.7` |
| `HELPER_LLM_MAX_TOKENS` | Optional max tokens, default `1000` |

If LLM configuration is missing or fails, Coordinator falls back to built-in rules and help retrieval.

## Coordinator Does Not Reply

1. Confirm `bot-helper-001` exists and is a channel member.
2. Confirm the message contains the exact username `@Coordinator`.
3. Check backend logs for LLM or Bot runtime errors.
4. If Agent Bridge Bots do not reply, confirm their providers are connected.
