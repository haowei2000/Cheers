# AgentNexus 安装部署说明

> **语言**：中文 | [English](安装部署说明.md)

> 面向**部署与运维人员**：环境要求、Docker 与本地两种安装方式、配置、数据库迁移、首次初始化。不涉及日常使用与管理操作。

---

## 一、环境要求

| 项目 | 要求 |
|------|------|
| 操作系统 | macOS、Linux；Windows 建议 WSL2 或等效环境 |
| Docker 方式 | Docker 20.10+、Docker Compose v2+ |
| 本地方式 | Python 3.13+、Node.js 18+、PostgreSQL 16+、Redis 7+ |
| 可选 | RustFS/MinIO 等 S3 兼容对象存储；kkFileView 文档预览服务 |
| 浏览器 | Chrome、Firefox、Safari、Edge 近期版本（访问前端用） |

---

## 二、安装方式一：Docker Compose（推荐）

### 2.1 启动服务

在项目根目录执行：

```bash
cp docker-compose.yml.template docker-compose.yml
cp .env.example .env

# 首次启动前至少修改 .env 中的 ADMIN_PASSWORD、JWT_SECRET_KEY、
# POSTGRES_PASSWORD、STORAGE_S3_ACCESS_KEY 和 STORAGE_S3_SECRET_KEY。
docker compose up -d
```

- **前端**：http://localhost:80
- **后端 API**：http://localhost:8000
- 数据默认写入项目根目录 `data/`，包括 PostgreSQL、RustFS、上传文件、日志等。

### 2.1.1 生产化安全启动（推荐）

公网部署建议使用 TLS 反向代理覆盖文件：

```bash
cp docker-compose.yml.template docker-compose.yml
cp .env.example .env

# 准备证书文件（域名证书）：
# ./certs/fullchain.pem
# ./certs/privkey.pem

docker compose -f docker-compose.yml -f docker-compose.production.tls.yml up -d
```

`.env.example` 建议至少配置：

- `APP_DOMAIN`：公网域名，例如 `agentnexus.example.com`
- `CORS_ALLOWED_ORIGINS`：如 `https://agentnexus.example.com`
- `STORAGE_S3_PUBLIC_ENDPOINT`：建议等于公网域名
- `STORAGE_S3_ACCESS_KEY` / `STORAGE_S3_SECRET_KEY`：统一使用这一套密钥（与 RustFS 共用）
- `HTTP_PORT=80`、`HTTPS_PORT=443`
- `TLS_CERT_DIR`、`TLS_CERT_FILE`、`TLS_KEY_FILE`：证书挂载与文件路径

说明：

- `docker-compose.production.tls.yml` 会启用 `tls-edge`（Caddy）并强制 `CORS_ALLOWED_ORIGINS` 为后端与 RustFS 生效来源；
- 默认把对外端口收敛到 `127.0.0.1`，仅通过 Caddy 提供 `HTTPS`；
- `PUBLIC_BASE_URL` 需与你的公开域名保持一致，供预览/下载回源使用。

### 2.2 数据库迁移（首次或升级后）

backend 容器启动时默认会执行主业务库和 Context Store 的 Alembic 迁移：

- `DB_MIGRATION_MODE=upgrade`
- `CONTEXT_DB_MIGRATION_MODE=upgrade`

如需手动执行，可使用工具服务：

```bash
docker compose --profile tools run --rm db-migrate
```

如果你只想检查当前 revision，可进入 backend 容器执行 Alembic 命令。

### 2.3 数据库版本回退（代码回滚前）

当数据库已经升级到较新的 Alembic revision，而你要回退到旧代码时，必须先使用**仍然包含当前 revision 文件的新镜像/新代码**把数据库降到旧代码认识的 revision。否则旧镜像启动时会报：

```text
Can't locate revision identified by '044'
```

推荐回退顺序：

```bash
# 1. 备份主库
docker compose exec -T postgres pg_dump \
  -U "${POSTGRES_USER:-agentnexus}" \
  "${POSTGRES_DB:-agentnexus}" > backup_before_rollback.sql

# 2. 停止业务 backend，避免回退时仍有写入
docker compose stop backend

# 3. 用当前新镜像执行一次性 downgrade，然后退出
# 示例：把主业务库从 044 降到 043；Context Store 不变
DB_MIGRATION_MODE=downgrade \
DB_MIGRATION_TARGET=043 \
CONTEXT_DB_MIGRATION_MODE=none \
ALLOW_DB_DOWNGRADE=1 \
docker compose --profile tools run --rm db-migrate

# 4. 再切换到旧代码/旧镜像并启动
git switch <old-branch-or-tag>
docker compose build backend
docker compose up -d backend
```

如果 Context Store 也需要回退，必须使用它自己的 revision：

```bash
DB_MIGRATION_MODE=none \
CONTEXT_DB_MIGRATION_MODE=downgrade \
CONTEXT_DB_MIGRATION_TARGET=<context_revision> \
ALLOW_DB_DOWNGRADE=1 \
docker compose --profile tools run --rm db-migrate
```

相关环境变量：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `DB_MIGRATION_MODE` | `upgrade` | 主业务库迁移模式：`upgrade`、`downgrade`、`none` |
| `DB_MIGRATION_TARGET` | `head` | 主业务库目标 revision；downgrade 时必须显式填写，如 `043` |
| `CONTEXT_DB_MIGRATION_MODE` | `upgrade` | Context Store 迁移模式：`upgrade`、`downgrade`、`none` |
| `CONTEXT_DB_MIGRATION_TARGET` | `head` | Context Store 目标 revision |
| `ALLOW_DB_DOWNGRADE` | `0` | 降级保险开关；执行 downgrade 必须设为 `1` |
| `MIGRATE_ONLY` | `1`（`db-migrate` 服务） | 只执行迁移，不启动 API 服务 |

> 注意：并非所有 downgrade 都是无损的。加/删索引通常安全；删除字段、拆表、清洗数据类迁移可能丢失数据，必须先备份并确认对应 migration 的 `downgrade()` 行为。

### 2.4 首次使用前初始化（二选一）

**方式 A：使用种子数据（推荐，便于开箱测试）**

`.env.example` / `docker-compose.yml.template` 默认 `SEED_DATA=1`，首次启动会自动创建默认工作空间、默认项目、一个默认提示词模板、内置统一 Bot 和管理员用户。若你关闭了自动种子，可手动执行：

```bash
docker compose exec backend python -m app.db.seed
```

种子数据会创建：默认工作空间、**默认项目**、一个默认提示词模板、**内置协作助手（@Coordinator）**、管理员用户。打开前端进入默认项目后，可输入 **@Coordinator 怎么用** 获取使用引导。

**方式 B：手动初始化**

1. **创建工作空间**：见 [系统管理说明书 - 创建工作空间](/manual/系统管理说明书#二如何创建工作空间与项目)。
2. **创建项目**：按系统管理说明书「创建项目」调用 API 创建至少一个项目。
3. 打开前端，应能在左侧看到项目列表。

## 三、安装方式二：本地安装（开发或非容器部署）

### 3.1 后端

本地开发仍然需要 PostgreSQL、Redis 和对象存储。最省事的方式是只启动依赖服务：

```bash
cp docker-compose.yml.template docker-compose.yml
cp .env.example .env
docker compose up -d postgres redis rustfs kkfileview
```

```bash
cd backend
uv venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
uv sync --extra dev
```

复制并编辑环境配置（见 §四）：

```bash
cp ../.env.example ../.env
```

启动后端：

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 3.2 前端

```bash
cd frontend
npm install
npm run dev
```

浏览器访问前端开发地址（如 http://localhost:5173）。若前端需请求后端，请配置代理或后端地址为 http://localhost:8000。

### 3.3 数据库迁移（首次或升级后）

在**项目根或 backend 目录**下（已激活虚拟环境）：

```bash
cd backend
uv run alembic upgrade head
uv run alembic -c alembic_context.ini upgrade head
```

### 3.4 本地数据库回退

本地非容器部署时，也要先在**包含当前 revision 文件的代码版本**下执行 downgrade，再切换旧代码：

```bash
cd backend
uv run alembic downgrade 043
# 如 Context Store 也需要回退：
uv run alembic -c alembic_context.ini downgrade <context_revision>
```

完成后再切换旧代码并启动后端。

### 3.5 首次使用前初始化（二选一）

**方式 A：使用种子数据（推荐）**

在项目根目录、已激活 backend 虚拟环境时执行：

```bash
cd backend
python -m app.db.seed
```

会创建默认工作空间、默认项目、一个默认提示词模板、内置协作助手（@Coordinator）等；或启动后端时设置环境变量 **SEED_DATA=1**，首次启动会自动执行种子。

**方式 B：手动初始化**

与 Docker 方式相同：先创建工作空间，再创建项目（见 [系统管理说明书](系统管理说明书.md)）。

## 四、配置说明

### 4.1 环境变量

在项目根目录创建 **.env**（可复制 **.env.example** 后修改）。常用项：

| 变量 | 说明 | 示例 |
|------|------|------|
| DATABASE_URL | 主业务库连接串 | `postgresql+asyncpg://agentnexus:change-me-postgres-password@localhost:5432/agentnexus` |
| CONTEXT_DB_URL | Context Store 连接串，默认可与主库相同 | `postgresql+asyncpg://agentnexus:change-me-postgres-password@localhost:5432/agentnexus` |
| REDIS_URL | Redis 地址（可选） | `redis://localhost:6379/0` |
| DATA_DIR | 数据根目录（上传、转换等） | `data` |
| STORAGE_BACKEND | 文件存储后端，Docker 默认 `s3` | `s3` |
| PUBLIC_BASE_URL | 外部可访问的 AgentNexus 地址，供文件预览服务回源 | `http://localhost` |
| KKFILEVIEW_BASE_URL | kkFileView 对外地址，启用文档预览时需要 | `http://localhost/preview` |
| FILE_RETENTION_DAYS | 文件保留天数，默认 365 天；设为 `0` 或负数表示不过期 | `365` |
| DEBUG | 调试模式 | `false` |

Docker 部署时上述变量可在 **docker-compose.yml** 的 **environment** 中覆盖。

> **本地 PostgreSQL 查询提醒**：排查主业务库或记忆数据时，必须先读取项目根目录或 `backend` 目录下的 **.env**。`memory_entries`、`history_pages`、文件索引等主库表按 **DATABASE_URL** 查询；旧版 `context_store` 或派生层回退按 **CONTEXT_DB_URL** 查询。不要直接套用文档示例里的库名、端口或账号。

### 4.2 钉钉登录（可选）

钉钉登录默认关闭。启用步骤：

1. 在钉钉开发者后台创建应用，记录 `ClientID` 与 `ClientSecret`。
2. 在钉钉安全设置中配置回调地址：

```text
<PUBLIC_BASE_URL>/api/v1/auth/dingtalk/callback
```

3. 申请委托权限：`openid`、`corpid`、`Contact.User.Read`。
4. 在 `.env` 中配置：

```env
DINGTALK_LOGIN_ENABLED=true
DINGTALK_CLIENT_ID=<client_id>
DINGTALK_CLIENT_SECRET=<client_secret>
DINGTALK_ALLOWED_CORP_IDS=dingxxxxxxxx
DINGTALK_DEFAULT_CORP_ID=dingxxxxxxxx
```

`DINGTALK_ALLOWED_CORP_IDS` 必填，多个 `corpId` 用英文逗号分隔。AgentNexus 会拒绝白名单外的钉钉组织；首次通过校验的钉钉用户会自动创建为普通 `member` 账号。

### 4.3 数据与路径说明

- Docker 默认把数据挂载到项目根目录 `data/`，便于备份和迁移。
- `DATA_DIR` 只控制后端上传、转换、日志等应用数据，不等同于 PostgreSQL 数据目录。
- PostgreSQL 数据目录由 `docker-compose.yml` 中的 `./data/postgres:/var/lib/postgresql/data` 控制。
- RustFS 数据目录由 `./data/rustfs:/data` 控制。

### 4.4 升级迁移建议（重要）

升级到新版本后，若出现 `no such column`（如 `channels.auto_assist`、`bot_accounts.description`），通常是数据库迁移历史与当前结构不一致。

建议顺序：

1. 先执行 `alembic upgrade head`
2. 若迁移中途因“表已存在/历史分叉”失败，先备份 PostgreSQL 数据库
3. 按 [技术排查Q&A](技术排查Q&A.md) 的数据库结构排查步骤核对缺失列，再补齐
4. 重启后端并验证 `/health` 与 `/api/v1/channels`

## 五、部署后检查

1. **健康检查**：访问 `http://localhost:8000/health`，应返回 `{"status":"ok"}`。
2. **API 文档**：访问 `http://localhost:8000/docs` 可查看 Swagger。
3. **前端**：访问前端地址，若已按系统管理说明书完成工作空间与项目初始化，左侧应显示项目列表。
