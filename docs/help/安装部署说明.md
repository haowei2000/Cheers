# AgentNexus Installation Guide

> **Language**: English | [中文](安装部署说明.zh-CN.md)

This guide covers environment requirements, Docker Compose deployment, local development, migrations, initial seed data, and deployment checks.

## Requirements

| Area | Requirement |
|---|---|
| OS | macOS or Linux. Windows users should use WSL2 or an equivalent Linux environment. |
| Docker deployment | Docker 20.10+ and Docker Compose v2+ |
| Local development | Python 3.13+, Node.js 18+, PostgreSQL 16+, Redis 7+ |
| Optional services | RustFS/MinIO-compatible object storage, kkFileView document preview |
| Browser | Recent Chrome, Firefox, Safari, or Edge |

## Docker Compose Deployment

```bash
cp docker-compose.yml.template docker-compose.yml
cp .env.example .env

# Change at least these values before first startup:
# ADMIN_PASSWORD, JWT_SECRET_KEY, POSTGRES_PASSWORD,
# STORAGE_S3_ACCESS_KEY, STORAGE_S3_SECRET_KEY
docker compose up -d
```

Default endpoints:

- Frontend: `http://localhost`
- Backend API: `http://localhost:8000`
- Swagger: `http://localhost:8000/docs`
- Health check: `http://localhost:8000/health`

### Production hardening (recommended for public deployment)

Use the overlay compose and reverse proxy:

```bash
cp docker-compose.yml.template docker-compose.yml
cp .env.example .env

# Put TLS certs:
# ./certs/fullchain.pem
# ./certs/privkey.pem

docker compose -f docker-compose.yml -f docker-compose.production.tls.yml up -d
```

Production defaults in `.env.example`:

- `APP_DOMAIN`: your public host, e.g. `agentnexus.example.com`
- `CORS_ALLOWED_ORIGINS`: `https://agentnexus.example.com`
- `STORAGE_S3_PUBLIC_ENDPOINT`: `https://agentnexus.example.com`
- `STORAGE_S3_ACCESS_KEY` / `STORAGE_S3_SECRET_KEY`: one canonical key pair for backend and RustFS
- `HTTP_PORT=80` and `HTTPS_PORT=443`
- `TLS_CERT_DIR`, `TLS_CERT_FILE`, `TLS_KEY_FILE`: certificate mount and file paths

Notes:

- `docker-compose.production.tls.yml` adds a `tls-edge` (Caddy) service and forces CORS to the configured `CORS_ALLOWED_ORIGINS`.
- This setup keeps backend/frontend/rustfs public ports bound to loopback by default and exposes HTTPS externally through Caddy.
- `PUBLIC_BASE_URL` should align with your public domain and can be set in `.env`.

Docker stores development data under `data/`, including PostgreSQL data, RustFS files, uploads, converted files, and logs.

## Database Migrations

The backend container runs Alembic migrations by default:

- `DB_MIGRATION_MODE=upgrade`
- `CONTEXT_DB_MIGRATION_MODE=upgrade`

Manual migration command:

```bash
docker compose --profile tools run --rm db-migrate
```

For local development:

```bash
cd backend
uv run alembic upgrade head
uv run alembic -c alembic_context.ini upgrade head
```

Downgrades must be run from a code version that still contains the current Alembic revision files. Always back up PostgreSQL before downgrading.

## Seed Data

With `SEED_DATA=1`, the backend creates a default workspace, default channel, administrator, one default prompt template, and the built-in Coordinator Bot. To run manually:

```bash
docker compose exec backend python -m app.db.seed
```

Then open the default channel and send:

```text
@Coordinator how do I use this?
```

## Local Development

Start dependencies:

```bash
cp docker-compose.yml.template docker-compose.yml
cp .env.example .env
docker compose up -d postgres redis rustfs kkfileview
```

Backend:

```bash
cd backend
uv venv
source .venv/bin/activate
uv sync --extra dev
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

Frontend:

```bash
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173` for the Vite development server.

## Important Environment Variables

| Variable | Purpose |
|---|---|
| `DATABASE_URL` | Main PostgreSQL database URL |
| `CONTEXT_DB_URL` | Context Store database URL; can point to the same database |
| `REDIS_URL` | Redis connection URL |
| `DATA_DIR` | Application data directory |
| `STORAGE_BACKEND` | `s3` for RustFS/MinIO-compatible storage |
| `PUBLIC_BASE_URL` | Public AgentNexus URL used by preview services |
| `KKFILEVIEW_BASE_URL` | kkFileView public preview URL |
| `JWT_SECRET_KEY` | Stable random JWT signing key; required for production |
| `FILE_RETENTION_DAYS` | File retention period; `0` or negative disables expiry |

## DingTalk Login

DingTalk login is disabled by default. To enable it:

1. Create a DingTalk app in the DingTalk developer console and record the `ClientID` and `ClientSecret`.
2. In DingTalk security settings, register the redirect URL as:

```text
<PUBLIC_BASE_URL>/api/v1/auth/dingtalk/callback
```

3. Grant delegated permissions for `openid`, `corpid`, and `Contact.User.Read`.
4. Configure AgentNexus:

```env
DINGTALK_LOGIN_ENABLED=true
DINGTALK_CLIENT_ID=<client_id>
DINGTALK_CLIENT_SECRET=<client_secret>
DINGTALK_ALLOWED_CORP_IDS=dingxxxxxxxx
DINGTALK_DEFAULT_CORP_ID=dingxxxxxxxx
```

`DINGTALK_ALLOWED_CORP_IDS` is mandatory and comma-separated. AgentNexus rejects DingTalk accounts outside this allowlist. New DingTalk users are auto-created as regular `member` accounts after the DingTalk identity and `corpId` pass validation.

## Post-deployment Checks

1. `docker compose ps` shows backend, frontend, postgres, redis, rustfs, and kkfileview running.
2. `curl http://localhost:8000/health` returns `{"status":"ok"}`.
3. `http://localhost:8000/docs` opens Swagger.
4. The frontend loads and can sign in, show channels, send messages, and mention `@Coordinator`.
5. File upload and preview work with your storage and preview configuration.
