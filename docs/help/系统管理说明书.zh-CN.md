# Cheers 系统管理说明书

> **语言**：中文 | [English](系统管理说明书.md)

> 面向**系统管理员**：如何创建工作空间与项目、如何添加/移除成员、如何把支持 ACP 的本地 Agent 接入为 Bot 等。OpenClaw 链接为废弃/遗留入口，新部署请优先使用 ACP 本地 Agent（通过 ACP Connector 接入）。需要会使用命令行与接口（API）。

---

## 管理员入口地址

管理员与普通用户使用同一前端；常见管理操作可在前端「管理」入口完成，复杂操作也可以通过**后端 API** 执行。Rust 网关没有 Swagger UI —— REST 路由定义见 `server/src/router.rs`。

| 入口 | 地址 |
|------|------|
| **聊天前端**（与用户相同） | Docker：`http://localhost:80`；本地开发：`http://localhost:5173` |
| **健康检查** | `http://localhost:8000/health` |
| **连接器发现（需登录鉴权）** | `GET http://localhost:8000/api/v1/ops/connector-discovery` —— 返回 ACP 连接器应拨号的 control/data WebSocket 地址 |

若后端端口或域名不同，请将上述 `localhost:8000` 替换为实际后端地址。

### 管理界面中的「表格表单」模式

当前管理操作（创建项目、创建 Bot、添加/移除成员等）均采用**同一套交互方式**：在「管理」弹窗中，**左列是字段说明/问题，右列是对应的输入框或操作按钮**，用户填写后点击提交即可完成。后续新增的系统设置（如更多工作空间配置、频道配置等）也会沿用这种「表格表单」方式，便于统一体验与扩展。

### 钉钉登录管理

启用钉钉登录后，用户可在登录弹窗点击 **Continue with DingTalk**。普通浏览器会走钉钉 OAuth；钉钉客户端内打开时，如果配置了默认或唯一允许的 `corpId`，会优先使用端内委托授权。

安全默认策略：

- 未设置 `DINGTALK_LOGIN_ENABLED=true` 时，钉钉登录不可用。
- 必须配置 `DINGTALK_ALLOWED_CORP_IDS`；白名单外组织会被拒绝。
- 首次钉钉登录只会自动创建普通 `member` 用户。
- 系统不保存钉钉 access token 或 refresh token。
- v1 不会按邮箱或手机号自动绑定已有密码账号。

## 一、概念说明

| 概念 | 说明 |
|------|------|
| **工作空间（Workspace）** | 顶层容器，相当于“组织”。创建项目前必须先有一个工作空间。 |
| **项目（频道 / Channel）** | 协作单元，成员在项目里聊天、@ Bot、发文件。 |
| **成员** | 分为 **user**（用户）和 **bot**（Bot）；均需被“加入”到具体项目后，才能在该项目中生效。 |
| **Bot 账号** | 每个支持 ACP 的本地 Agent、OpenClaw 遗留实例或其它 AI 服务在系统里对应一个 Bot 账号；注册 Bot 后再把它加入项目，用户才能 @ 它。 |

## 二、如何创建工作空间与项目

### 2.1 创建工作空间（只需做一次）

**【推荐】前端表格表单**：在「管理」→「创建工作空间」中填写「空间名称」后点击「创建」即可。

**或**调用 API：

```bash
curl -X POST http://localhost:8000/api/v1/workspaces \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <你的访问令牌>" \
  -d '{"name":"默认空间"}'
```

记下返回值中的 **`workspace_id`**，创建项目时要用。

### 2.2 创建项目（频道）

在已有工作空间下创建项目，调用 API（基础地址默认 `http://localhost:8000`，下同）：

```bash
curl -X POST http://localhost:8000/api/v1/channels \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <你的访问令牌>" \
  -d '{"workspace_id":"ws-default-001","name":"我的第一个项目","type":"public","purpose":"日常协作"}'
```

将 **`ws-default-001`** 换成你的工作空间 ID，**name** 可改为任意项目名称。
返回的 JSON 中的 **`channel_id`** 即为项目 ID，后续加人、加 Bot 时需用到。

## 三、如何管理项目成员（加人 / 加 Bot / 移除）

### 3.1 查看项目成员

```bash
curl "http://localhost:8000/api/v1/channels/项目ID/members" \
  -H "Authorization: Bearer <你的访问令牌>"
```

将 **项目ID** 替换为实际的 `channel_id`。

### 3.2 添加成员（用户或 Bot）

**方式一：聊天内邀请 Bot（Slack 风格，推荐）**

- 用户在频道内输入 **@** 时，会显示全部 Bot 列表（已在频道的排前，未加入的灰色排后）。
- 若选择未加入的 Bot，系统提示「是否邀请加入？」；用户确认后，Bot 自动加入该频道，输入框插入 @Bot名，不打断提问流程。
- 无需去「管理」填 ID，适合快速拉 Bot 进群。详见《普通用户使用说明》§3.2.1。

**方式二：管理端添加**

```bash
curl -X POST "http://localhost:8000/api/v1/channels/项目ID/members" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <你的访问令牌>" \
  -d '{"member_id":"用户或Bot的ID","member_type":"user"}'
```

- 添加**用户**：`member_type` 填 **`user`**，`member_id` 填该用户的 **user_id**（需先在 users 表存在）。
- 添加**Bot**：`member_type` 填 **`bot`**，`member_id` 填该 Bot 的 **bot_id**（需先在 bot_accounts 表存在，见下文「四、Agent Bridge 接入」）。

### 3.3 移除成员

```bash
curl -X DELETE "http://localhost:8000/api/v1/channels/项目ID/members/成员ID" \
  -H "Authorization: Bearer <你的访问令牌>"
```

**成员ID** 即该用户或 Bot 的 user_id / bot_id。

## 四、如何让 Agent Bridge provider 接入（注册 Bot 并加入项目）

“接入”指：把支持 ACP 的本地 Agent 通过 ACP Connector 注册为 Cheers 里的一个 Bot，用户在某项目里 @ 该 Bot 时，由外部 provider 回复。OpenClaw provider 仅保留为废弃/遗留路径。

### 4.1 适配器模式说明

Cheers 当前只保留两类 Bot 绑定：

| binding_type | 适配器 | 说明 |
|------|--------|------|
| `http` | HTTP Bot | 后端按 Bot 绑定的 AIModel + PromptTemplate 直接调用 OpenAI 兼容模型。 |
| `agent_bridge` | Agent Bridge Bot | 外部 provider 通过 `/ws/agent-bridge/control` 与 `/ws/agent-bridge/data` 反向连接，异步回推 reply / delta / done。推荐使用 ACP Connector；OpenClaw provider 为废弃/遗留入口。 |

平台没有内置 bot —— 旧的 `Coordinator`（`bot-helper-001`）已随 Python 后端移除。所有 Bot 都由管理员或用户创建，背后是外部智能体或模型。

### 4.2 第 1 步：注册 Bot 账号

**【推荐】通过管理后台 UI：**

1. 打开前端 → 左侧「管理」→ **Bot 管理** 区块
2. 点击新建 Bot，选择 **Agent Bridge Bot**
3. 填写 username、显示名称、可选的 Provider agent id 与 Prompt 模板
4. 创建后复制一次性 `agb_...` token，并写入 provider 配置

如需以脚本方式为运行智能体的主机完成接入，请使用一次性 **enrollment code** 流程（见下文「五、如何让外部主机自动完成接入」）。旧的 `POST /docs/agent-bridge/register` 接口属于已删除的 Python 后端，已不存在。

### 4.2.1 关闭 Agent Bridge 会话

在 Bot 管理的 Status 页可以查看 Agent Bridge Bot 的所有会话。管理员可关闭不再使用的会话；关闭后会话状态变为 `closed`，活跃 scope 绑定会被脱离，历史记录仍保留用于审计和排查。也可以调用：

```bash
DELETE /api/v1/channels/{channel_id}/bots/{bot_id}/sessions/{session_id}
```

### 4.3 第 2 步：将 Bot 加入项目

**【推荐】通过管理后台 UI：**

1. 左侧「管理」→ **Bot 与频道** 区块
2. 选择目标项目，点击「添加成员」，在复选框列表中勾选 Bot，点击「添加选中」

**或**调用 API：

```bash
curl -X POST "http://localhost:8000/api/v1/channels/项目ID/members" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <你的访问令牌>" \
  -d '{"member_id":"你的BOT_ID","member_type":"bot"}'
```

完成后，用户在该项目内 @ 该 Bot 的 **username**（如 `@mybot`）即可触发。

### 4.4 Agent Bridge 调用协议

Cheers 通过两条 WebSocket 与 provider 通信：

1. **control**：provider 使用 `Authorization: Bearer agb_...` 连接 `/ws/agent-bridge/control`，接收 membership 快照和频道加入/移除事件。
2. **data**：provider 连接 `/ws/agent-bridge/data`，接收 `message` 任务，回推 `reply`、`delta`、`done`、`error`。
3. **落盘广播**：后端统一通过 Message DTO 写入消息并广播 `message_done` / `message_stream`。

ACP Connector（`packages/cheers-acp-connector-rs`）是推荐的 Agent Bridge provider 实现。详见 [Agent Bridge 接入指南](AgentBridge接入指南.zh-CN.md) 与 [本地Bot配置指南](本地Bot配置指南.zh-CN.md)。OpenClaw 插件为废弃/遗留入口。

### 4.5 HTTP Bot 模式说明

HTTP Bot 不再使用 `Agent Bridge 配置`，而是选择一个 AIModel 和 PromptTemplate。后端按 OpenAI 兼容 Chat Completions 协议调用对应模型。

## 五、如何让外部主机自动完成接入（enrollment code）

本节说明：**如何让运行智能体的主机以脚本方式完成接入**。管理员先在 Cheers 里创建 Agent Bridge Bot，再为它签发一个一次性 enrollment code；目标主机用这个 code 换取 bot token 和连接器配置。以下路径均已对照 `server/src/router.rs` 核实。

### 5.1 相关接口

| 步骤 | 方法 | 路径（基础地址默认 `http://localhost:8000`） |
|------|------|---------------------------------------------|
| 签发一次性 enrollment code | POST | `/api/v1/bots/{bot_id}/enrollment` —— 需要 Bot 拥有者/管理员 JWT。明文 code **只返回一次**；单次有效，15 分钟过期。对同一路径 `DELETE` 可吊销该 Bot 的全部未用 code。 |
| 在目标主机上兑换 code | POST | `/api/v1/enrollment/redeem` —— 公开接口，由 code 本身鉴权。原子地占用 code、轮换 bot token，并返回 token 和可直接运行的连接器配置。 |
| 一行命令安装连接器 | GET | `/api/v1/install.sh` —— 公开接口。返回内嵌 API 地址的安装脚本；脚本从 `CHEERS_ENROLL_CODE` 环境变量读取 code 并自行兑换。 |
| 获取安装引导 | GET | `/api/v1/enrollment/guidance` —— 需登录鉴权。返回安装 URL 和一段可直接交给智能体的提示词模板。 |
| 获取连接器配置（手动模式） | GET | `/api/v1/bots/{bot_id}/connector-config` —— Bot 拥有者/管理员。返回从 sidecar 文件读取 token 的配置；token 需另行通过 `POST /api/v1/bots/{bot_id}/token` 签发。 |
| 连接器发现 | GET | `/api/v1/ops/connector-discovery` —— 需登录鉴权。返回连接器应拨号的 control/data WebSocket 地址。 |

### 5.2 接入流程概览

1. **创建 Bot**：管理员在「管理」→「Bot 管理」创建 Agent Bridge Bot（见 4.2）。
2. **签发 code**：调用 `POST /api/v1/bots/{bot_id}/enrollment`，把返回的明文 code 交给目标主机（15 分钟内有效，只能用一次）。
3. **主机兑换**：目标主机以 `CHEERS_ENROLL_CODE=<code>` 运行 `GET /api/v1/install.sh` 下载的安装脚本，或直接调用 `POST /api/v1/enrollment/redeem`；成功后获得轮换过的 bot token 和可直接运行的连接器配置。token 只明文返回一次，必须立即保存；丢失后需重新签发 code 或通过 `POST /api/v1/bots/{bot_id}/token` 轮换。
4. **连接器上线**：连接器用 token 连接 `/ws/agent-bridge/control` 与 `/ws/agent-bridge/data`（实际地址可通过 `GET /api/v1/ops/connector-discovery` 查询）。

> 旧的 `/docs/agent-bridge/discovery`、`/docs/agent-bridge/help`、`/docs/agent-bridge/register`、`/docs/agent-bridge/auth-check` 接口，以及账号密码直连注册和 OpenClaw 插件下载，均属于已删除的 Python 后端，已不存在。

## 六、管理操作速查

| 操作 | 方法 | 路径 / 请求体 |
|------|------|----------------|
| 获取所有项目列表 | GET | `/api/v1/channels` |
| 创建项目 | POST | `/api/v1/channels`，body：`workspace_id`, `name`, `type`（如 public）, `purpose`（可选） |
| 获取某项目成员 | GET | `/api/v1/channels/{项目ID}/members` |
| 添加成员（用户或 Bot） | POST | `/api/v1/channels/{项目ID}/members`，body：`member_id`, `member_type`（user / bot） |
| 移除成员 | DELETE | `/api/v1/channels/{项目ID}/members/{成员ID}` |
| 创建工作空间 | POST | `/api/v1/workspaces`，body：`name` |
| 签发 Bot enrollment code | POST | `/api/v1/bots/{bot_id}/enrollment` |
| 兑换 enrollment code | POST | `/api/v1/enrollment/redeem` |
| 连接器发现 | GET | `/api/v1/ops/connector-discovery` |

所有接口基础路径：`http://localhost:8000`（或实际后端地址）。网关没有 Swagger UI；更多参数与响应以 `server/src/router.rs` 中定义的路由为准。

### Bot 无反应时如何排查

1. **确认该 Bot 存在、已启用且已加入当前项目**
   平台没有内置 bot；只有管理员/用户创建并加入频道的 Bot 才能被 @ 触发。未加入时先按「三、3.2」把它加进项目。

2. **确认 @ 的名字与 Bot 的 username 完全一致**
   路由是确定性的「@提及 → bot」查找，不能多空格或错字；没有 @ 任何人的消息不会触发任何 Bot。

3. **Agent Bridge Bot 需确认 provider/连接器在线**
   查看 Bot Status 面板，或调用 `GET /api/v1/bots/{bot_id}/status`。

4. **查看网关日志**
   关注 Bot 运行时或 provider 相关错误。

### 日志文件（便于排查错误）

后端会将运行日志写入文件（需配置 `LOG_DIR`，默认 `data/logs`，相对 backend 目录）：

| 文件 | 内容 |
|------|------|
| `cheers.log` | 通用日志（INFO 及以上），按大小轮转 |
| `error.log` | 仅错误与异常（ERROR 及以上），便于快速排查 |

未设置 `LOG_DIR` 或目录不可写时，仅输出到控制台。环境变量：`LOG_DIR`、`LOG_MAX_BYTES`（单文件最大字节，默认 5MB）、`LOG_BACKUP_COUNT`（保留份数，默认 3）。未捕获异常会记录完整堆栈到上述日志并返回 500。

### 连接器版本与自动更新

网关可以通告一个连接器发行版本，供**签名自动更新**使用（连接器 >= 0.1.27）：

1. 在网关环境里设置 `CHEERS_CONNECTOR_RELEASE_VERSION`（如 `0.1.27`）并重建容器。
   该版本会出现在 Agent Bridge hello 里，并经 `/api/v1/connector/download/…` 代理下载。
2. `GET /api/v1/bots/{bot_id}/status` 会返回每个 bot 的 `connector_version`、
   `latest_connector_version` 与 `update_available`。
3. 已开启 `[update] auto = true` 的宿主机会自动升级：连接器用编译进二进制的
   ed25519 公钥验签发行清单、校验各二进制 sha256、排空进行中的对话后原子替换并
   原地重启；未开启的宿主机则在日志与 `status` 命令里收到手动升级提醒。

发版方式：打 `connector-v<版本>` tag；CI 的 manifest 任务用仓库 secret
`CONNECTOR_SIGNING_KEY` 给更新清单签名——secret 缺失则该任务失败，
自动更新的连接器会忽略这次发布。
